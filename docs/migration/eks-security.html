EKS 보안EKS Security

On-Prem Kubernetes RBAC 기반 보안에서 AWS IAM 통합 보안 모델로 전환합니다. IRSA/Pod Identity로 세밀한 권한 관리를 구현하고, Pod Security Standards로 워크로드를 보호합니다.Transition from On-Prem Kubernetes RBAC-based security to AWS IAM integrated security model. Implement fine-grained permission management with IRSA/Pod Identity and protect workloads with Pod Security Standards.

IRSA & Pod IdentityIRSA & Pod Identity

IRSA & EKS Pod Identity - Pod 레벨 AWS 권한 관리IRSA & EKS Pod Identity - Pod-Level AWS Permission Management Security

On-Prem에서는 Kubernetes ServiceAccount만으로 권한을 관리했지만, EKS에서는 IRSA(IAM Roles for Service Accounts)를 통해 Pod에 AWS IAM Role을 직접 연결합니다. 노드 역할 대신 Pod 단위로 최소 권한 원칙을 적용할 수 있습니다.While On-Prem managed permissions with Kubernetes ServiceAccount alone, EKS uses IRSA (IAM Roles for Service Accounts) to directly associate AWS IAM Roles with Pods. Apply least privilege principle at Pod level instead of node role.

IRSA 설정 (Terraform)IRSA Setup (Terraform)

# IRSA IAM Role with OIDC Trust Policy
resource "aws_iam_role" "s3_reader" {
  name = "eks-s3-reader-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Principal = {
        Federated = aws_iam_openid_connect_provider.eks.arn
      }
      Action = "sts:AssumeRoleWithWebIdentity"
      Condition = {
        StringEquals = {
          "${local.oidc_provider}:sub" = "system:serviceaccount:production:s3-reader-sa"
          "${local.oidc_provider}:aud" = "sts.amazonaws.com"
        }
      }
    }]
  })
}

resource "aws_iam_role_policy_attachment" "s3_read" {
  role       = aws_iam_role.s3_reader.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess"
}hcl

IRSA에서 Pod Identity로 마이그레이션Migration from IRSA to Pod Identity 기존 IRSA를 Pod Identity로 전환할 때는 IAM Role Trust Policy에 두 방식을 모두 허용하도록 설정하여 무중단 마이그레이션이 가능합니다. 전환 완료 후 IRSA annotation을 제거합니다.When migrating from IRSA to Pod Identity, configure the IAM Role Trust Policy to allow both methods for zero-downtime migration. Remove IRSA annotations after migration is complete.

Pod Security StandardsPod Security Standards

Pod Security Standards (PSS) - 워크로드 보안 강화Pod Security Standards (PSS) - Strengthen Workload Security Security

Kubernetes 1.25에서 PodSecurityPolicy(PSP)가 제거되고 Pod Security Standards(PSS)/Pod Security Admission(PSA)으로 대체되었습니다. 네임스페이스 레이블로 간단하게 보안 정책을 적용할 수 있습니다.PodSecurityPolicy (PSP) was removed in Kubernetes 1.25 and replaced with Pod Security Standards (PSS)/Pod Security Admission (PSA). Security policies can be easily applied with namespace labels.

3가지 보안 레벨Three Security Levels

레벨Level 설명Description 사용 사례Use Case
Privileged 제한 없음 - 모든 권한 허용No restrictions - all privileges allowed CNI, CSI 드라이버, 모니터링 에이전트CNI, CSI drivers, monitoring agents
Baseline 알려진 권한 상승 방지Prevent known privilege escalations 일반 애플리케이션General applications
Restricted 최소 권한 원칙 강제Enforce least privilege principle 보안 민감 워크로드, 멀티테넌트Security-sensitive workloads, multi-tenant

네임스페이스 레이블로 PSA 적용Apply PSA with Namespace Labels

apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    # Enforce: 위반 시 Pod 생성 차단
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/enforce-version: v1.35
    # Audit: 위반 시 감사 로그 기록
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: v1.35
    # Warn: 위반 시 경고 메시지 표시
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: v1.35yaml

Restricted 레벨 Pod 예시Restricted Level Pod Example

apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: myapp:latest
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop:
            - ALL
      volumeMounts:
        - name: tmp
          mountPath: /tmp
  volumes:
    - name: tmp
      emptyDir: {}yaml
Baseline 이상 강제 시 호환성 확인Check Compatibility When Enforcing Baseline or Above 기존 워크로드가 hostNetwork, privileged container 등을 사용하는 경우 Baseline 적용 시 Pod 생성이 실패합니다. 먼저 warn 모드로 적용하여 위반 사항을 확인한 후 점진적으로 enforce로 전환하세요.If existing workloads use hostNetwork, privileged containers, etc., Pod creation will fail with Baseline enforcement. Apply warn mode first to identify violations, then gradually transition to enforce.

Secrets ManagementSecrets Management

AWS Secrets Manager + External Secrets OperatorAWS Secrets Manager + External Secrets Operator Security

Kubernetes Secret은 etcd에 Base64 인코딩으로 저장되어 보안에 취약합니다. AWS Secrets Manager와 External Secrets Operator를 사용하여 시크릿을 중앙 관리하고 자동 동기화합니다.Kubernetes Secrets are stored in etcd with Base64 encoding, which is insecure. Use AWS Secrets Manager with External Secrets Operator for centralized management and automatic sync.

External Secrets Operator 설정External Secrets Operator Setup

apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: aws-secrets-manager
spec:
  provider:
    aws:
      service: SecretsManager
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa
            namespace: external-secrets
---
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
  namespace: production
spec:
  refreshInterval: 1h  # 자동 동기화 주기
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: db-secret
  data:
    - secretKey: username
      remoteRef:
        key: production/database
        property: username
    - secretKey: password
      remoteRef:
        key: production/database
        property: passwordyaml

자동 로테이션 설정Automatic Rotation Setup

# AWS Secrets Manager에서 자동 로테이션 활성화
aws secretsmanager rotate-secret \
  --secret-id production/database \
  --rotation-lambda-arn arn:aws:lambda:us-east-1:180294183052:function:SecretsRotation \
  --rotation-rules '{"ScheduleExpression": "rate(30 days)"}'bash
시크릿 관리 도구 비교Secrets Management Tool Comparison AWS Secrets Manager + ESO: AWS 네이티브 통합, 자동 로테이션, IRSA 연동에 최적. HashiCorp Vault: 멀티 클라우드, 고급 정책, 동적 시크릿이 필요한 경우. Sealed Secrets: GitOps에서 암호화된 시크릿을 Git에 저장해야 할 때.AWS Secrets Manager + ESO: Best for AWS native integration, auto-rotation, IRSA. HashiCorp Vault: For multi-cloud, advanced policies, dynamic secrets. Sealed Secrets: When encrypted secrets need to be stored in Git for GitOps.

ECR 이미지 스캐닝 및 서명 검증ECR Image Scanning and Signature Verification Security

컨테이너 이미지의 취약점을 사전에 탐지하고, 서명된 이미지만 배포를 허용하여 Supply Chain 보안을 강화합니다.Detect vulnerabilities in container images proactively and strengthen Supply Chain security by allowing only signed images to deploy.
