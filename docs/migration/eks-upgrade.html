EKS 업그레이드 전략EKS Upgrade Strategy

온프레미스 kubeadm 기반 업그레이드와 EKS 관리형 업그레이드의 차이점을 이해하고, In-place 업그레이드와 Blue/Green 클러스터 전략을 상황에 맞게 선택하는 방법을 다룹니다.Understand the differences between on-premises kubeadm-based upgrades and EKS managed upgrades, and learn how to choose between In-place upgrades and Blue/Green cluster strategies based on your situation.

핵심 차이점Key Difference On-Prem에서는 Control Plane을 직접 kubeadm으로 업그레이드해야 하지만, EKS에서는 AWS가 Control Plane을 관리합니다. 단, EKS도 한 번에 한 마이너 버전만 업그레이드 가능합니다 (예: 1.29 → 1.30 → 1.31).On-Prem requires manual Control Plane upgrade via kubeadm, while EKS has AWS manage the Control Plane. However, EKS also only allows upgrading one minor version at a time (e.g., 1.29 → 1.30 → 1.31).

In-place 업그레이드In-place Upgrade

EKS In-place 업그레이드 절차EKS In-place Upgrade Procedure Operations

In-place 업그레이드는 기존 클러스터를 유지하면서 Control Plane → Add-ons → Node Groups 순서로 순차 업그레이드합니다. 다운타임을 최소화하면서 비용 효율적인 업그레이드가 가능합니다.In-place upgrade maintains the existing cluster while sequentially upgrading Control Plane → Add-ons → Node Groups. This enables cost-effective upgrades while minimizing downtime.

업그레이드 순서Upgrade Order

순서Order 컴포넌트Component 소요 시간Duration 다운타임Downtime
1 Control Plane ~15-25 min 없음 (API 일시 불가)None (API briefly unavailable)
2 vpc-cni ~5 min 없음None
3 coredns ~3 min 없음None
4 kube-proxy ~3 min 없음None
5 ebs-csi / efs-csi ~5 min 없음None
6 Node Groups ~30-60 min Rolling (PDB 준수)Rolling (PDB respected)

Control Plane 업그레이드Control Plane Upgrade

# 현재 클러스터 버전 확인
aws eks describe-cluster \
  --name multi-region-mall \
  --query "cluster.version" \
  --output text

# Control Plane 업그레이드 (1.34 → 1.35)
aws eks update-cluster-version \
  --name multi-region-mall \
  --kubernetes-version 1.35

# 업그레이드 상태 확인
aws eks describe-update \
  --name multi-region-mall \
  --update-id <update-id> \
  --query "update.status"

# 업그레이드 완료 대기
aws eks wait cluster-active --name multi-region-mallbash

Add-on 업그레이드Add-on Upgrade

# 호환 가능한 Add-on 버전 확인
aws eks describe-addon-versions \
  --addon-name vpc-cni \
  --kubernetes-version 1.35 \
  --query "addons[0].addonVersions[0:3].addonVersion"

# vpc-cni 업그레이드
aws eks update-addon \
  --cluster-name multi-region-mall \
  --addon-name vpc-cni \
  --addon-version v1.21.1-eksbuild.1 \
  --resolve-conflicts PRESERVE

# coredns 업그레이드
aws eks update-addon \
  --cluster-name multi-region-mall \
  --addon-name coredns \
  --addon-version v1.13.2-eksbuild.1 \
  --resolve-conflicts PRESERVE

# kube-proxy 업그레이드
aws eks update-addon \
  --cluster-name multi-region-mall \
  --addon-name kube-proxy \
  --addon-version v1.35.0-eksbuild.1 \
  --resolve-conflicts PRESERVEbash
Add-on 호환성 매트릭스Add-on Compatibility Matrix 각 EKS 버전에는 호환되는 Add-on 버전이 정해져 있습니다. aws eks describe-addon-versions로 반드시 호환 버전을 확인하세요. eksbuild suffix는 리전별로 다를 수 있습니다.Each EKS version has specific compatible Add-on versions. Always verify compatible versions with aws eks describe-addon-versions. The eksbuild suffix may differ by region.

Blue/Green 클러스터 업그레이드Blue/Green Cluster Upgrade

Blue/Green 클러스터 전환 전략Blue/Green Cluster Transition Strategy Infrastructure

Blue/Green 업그레이드는 새 클러스터를 생성하고, ArgoCD로 워크로드를 동기화한 후, 트래픽을 점진적으로 전환합니다. 롤백이 즉시 가능하며, 대규모 버전 점프나 미션 크리티컬 환경에 적합합니다.Blue/Green upgrade creates a new cluster, syncs workloads via ArgoCD, then gradually shifts traffic. Rollback is instant, making it suitable for major version jumps or mission-critical environments.

Blue/Green 단계별 체크리스트Blue/Green Step-by-Step Checklist

단계Phase 작업Task 검증 항목Verification
1. 준비Prepare Green 클러스터 Terraform 작성Write Green cluster Terraform VPC, Subnet, SG 공유 확인Verify VPC, Subnet, SG sharing
2. 생성Create Green 클러스터 + Add-ons 배포Deploy Green cluster + Add-ons kubectl get nodes Ready
3. 동기화Sync ArgoCD에 Green 클러스터 등록Register Green cluster in ArgoCD 모든 App Synced/HealthyAll Apps Synced/Healthy
4. 검증Validate Smoke Test, 내부 트래픽 테스트Smoke Test, internal traffic test Health check 200 OKHealth check 200 OK
5. 전환Switch NLB/ALB 타겟 그룹 가중치 전환NLB/ALB target group weight shift 10% → 50% → 100% 점진적10% → 50% → 100% gradual
6. 정리Cleanup Blue 클러스터 해제 (1주 후)Decommission Blue cluster (after 1 week) 비용 절감 확인Verify cost savings

Green 클러스터 생성 (Terraform)Create Green Cluster (Terraform)

resource "aws_eks_cluster" "green" {
  name     = "multi-region-mall-green"
  version  = "1.35"
  role_arn = aws_iam_role.cluster.arn

  vpc_config {
    subnet_ids              = var.private_subnet_ids  # 기존 VPC 공유
    endpoint_private_access = true
    endpoint_public_access  = true
    security_group_ids      = [aws_security_group.cluster.id]
  }

  # Karpenter용 태그
  tags = {
    "karpenter.sh/discovery" = "multi-region-mall-green"
  }
}hcl

멀티 리전 Blue/Green 팁Multi-Region Blue/Green Tip 멀티 리전 환경에서는 Secondary 리전(us-west-2)을 먼저 Blue/Green으로 전환하세요. 문제 발생 시 Route53 가중치 라우팅으로 Primary로 100% 전환하여 즉시 롤백 가능합니다.In multi-region environments, switch the Secondary region (us-west-2) first via Blue/Green. If issues occur, you can instantly rollback by shifting Route53 weighted routing 100% to Primary.

Pod Disruption BudgetPod Disruption Budget

PDB로 업그레이드 중 가용성 보장Ensuring Availability During Upgrades with PDB Operations

PodDisruptionBudget(PDB)은 노드 드레인 시 최소 가용 Pod 수를 보장합니다. 업그레이드 중에도 서비스 가용성을 유지하려면 모든 프로덕션 워크로드에 PDB를 설정하세요.PodDisruptionBudget (PDB) ensures minimum available pods during node draining. Set PDB for all production workloads to maintain service availability during upgrades.

PDB YAML 예시PDB YAML Example

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-gateway-pdb
  namespace: platform
spec:
  minAvailable: 2           # 최소 2개 Pod 유지
  selector:
    matchLabels:
      app: api-gateway
---
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: order-service-pdb
  namespace: fulfillment
spec:
  maxUnavailable: 1        # 최대 1개만 중단 허용
  selector:
    matchLabels:
      app: order-serviceyaml
minAvailable vs maxUnavailable 선택 기준Choosing minAvailable vs maxUnavailable
  • minAvailable: 레플리카 수가 고정된 경우 (예: 항상 3개 중 2개 이상 필요)minAvailable: When replica count is fixed (e.g., need at least 2 of 3)
  • maxUnavailable: 레플리카가 오토스케일링되는 경우 (HPA 사용 시 권장)maxUnavailable: When replicas autoscale (recommended with HPA)
  • 주의: minAvailable: 100%는 드레인을 완전히 차단합니다!Warning: minAvailable: 100% completely blocks draining!

PDB 상태 확인Check PDB Status

# 모든 PDB 상태 확인
kubectl get pdb -A

# 드레인 차단 PDB 찾기 (disruptionsAllowed=0)
kubectl get pdb -A -o json | jq -r '
  .items[] |
  select(.status.disruptionsAllowed == 0) |
  "\(.metadata.namespace)/\(.metadata.name): BLOCKED"
'bash

멀티 리전 업그레이드 순서Multi-Region Upgrade Order

Secondary → Primary 순차 업그레이드Sequential Upgrade: Secondary → Primary Architecture

멀티 리전 환경에서는 반드시 Secondary 리전을 먼저 업그레이드합니다. 트래픽 비중이 낮은 리전에서 충분히 검증한 후 Primary를 업그레이드하여 리스크를 최소화합니다.In multi-region environments, always upgrade the Secondary region first. Validate thoroughly in the lower-traffic region before upgrading Primary to minimize risk.

리전간 업그레이드 체크리스트Cross-Region Upgrade Checklist

  1. Secondary (us-west-2) 업그레이드Upgrade Secondary (us-west-2)
    • Control Plane → Add-ons → Node Groups 순서Order: Control Plane → Add-ons → Node Groups
    • Route53 가중치: us-west-2 10%, us-east-1 90% (선택)Route53 weights: us-west-2 10%, us-east-1 90% (optional)
  2. 24시간 안정화 관찰24-hour stabilization observation
    • 에러율, 레이턴시, Pod 재시작 모니터링Monitor error rate, latency, pod restarts
    • CloudWatch Alarms, Grafana 대시보드 확인Check CloudWatch Alarms, Grafana dashboards
  3. Primary (us-east-1) 업그레이드Upgrade Primary (us-east-1)
    • Secondary 검증 완료 후에만 진행Proceed only after Secondary validation
    • 업무 시간 외 (주말 또는 야간) 권장Recommended outside business hours (weekend or night)
  4. Add-on 버전 동기화 확인Verify Add-on version sync
    • 양 리전 Add-on 버전 일치 확인Confirm Add-on versions match in both regions
    • eksbuild suffix는 리전별로 다를 수 있음eksbuild suffix may differ by region
롤백 전략Rollback Strategy Secondary 업그레이드 중 문제 발생 시, Route53 가중치를 us-east-1 100%로 즉시 전환하세요. EKS Control Plane은 다운그레이드가 불가능하므로, 심각한 문제 시 새 클러스터 생성이 필요합니다.If issues occur during Secondary upgrade, immediately shift Route53 weights to 100% us-east-1. EKS Control Plane cannot be downgraded, so serious issues require creating a new cluster.
