CCS 데이터 아키텍처CCS Data Architecture

CCDI(Connected Car Data Interface)는 모든 차량 텔레메트리의 데이터 수집 및 처리 계층입니다. 이 섹션에서는 현재 및 목표 아키텍처, 데이터 티어링, Gen2/Gen3 처리 차이점, 개인정보 관리에 대해 다룹니다.The CCDI (Connected Car Data Interface) is the data collection and processing layer for all vehicle telemetry. This section covers the current and target architecture, data tiering, Gen2/Gen3 processing differences, and personal information management.

CCDI 아키텍처CCDI Architecture

CCDI 아키텍처 — AS-IS vs TO-BECCDI Architecture — AS-IS vs TO-BE Architecture

CCDI(Connected Car Data Interface)는 차량 텔레메트리 데이터를 수집, 처리, 제공하는 핵심 데이터 계층입니다. 커넥티드 차량에서 들어오는 모든 신호를 처리하고 처리된 데이터를 다운스트림 서비스에 배포합니다.CCDI (Connected Car Data Interface) is the central data layer responsible for collecting, processing, and serving vehicle telemetry data. It handles all incoming signals from connected vehicles and distributes processed data to downstream services.

데이터 티어링 전략Data Tiering Strategy

데이터 티어링 전략 — Hot/Warm/ColdData Tiering Strategy — Hot/Warm/Cold Database

데이터는 액세스 패턴과 수명에 따라 세 가지 티어를 통해 흐릅니다. 두 가지 병렬 라이프사이클이 작동합니다: DB 보존(3일)과 S3 아카이브(영구).Data flows through three tiers based on access patterns and age. Two parallel lifecycles operate: DB retention (3 days) and S3 archive (permanent).

VSS (Vehicle Signal Specification)

VSS는 차량 신호에 대한 표준화된 데이터 모델입니다. 가장 큰 데이터 볼륨을 나타내며 실시간 애플리케이션에 가장 낮은 지연 시간이 필요합니다.VSS is the standardized data model for vehicle signals. It represents the largest data volume and requires the lowest latency for real-time applications.

개인정보 관리Personal Information Management

개인정보(PI) 및 VIN 관리Personal Information (PI) & VIN Management Operations

Kafka 메시지에는 VIN(차량 식별 번호)이 기본 식별자로 포함됩니다. VIN이 미국 규정상 개인정보에 해당하는지는 법률 확인이 필요하지만, 아키텍처는 개인정보 준수 데이터 처리를 지원해야 합니다.Kafka messages contain VIN (Vehicle Identification Number) as the primary identifier. Whether VIN constitutes PI under US regulations requires legal confirmation, but the architecture must support PI-compliant data handling regardless.

VIN ID 기반 관리 프레임워크VIN ID-Based Management Framework

제안된 통합 접근 방식 (한국/미국)Proposed Unified Approach (Korea/US)
  1. VIN 토큰화Tokenization
    Raw VIN → 토큰화 서비스 → Token ID. 모든 다운스트림 처리에서 Token ID 사용. VIN-to-Token 매핑은 보안 볼트에 저장. Raw VIN → Tokenization Service → Token ID. Token ID used in all downstream processing. VIN-to-Token mapping stored in secured vault.
  2. 데이터 분리Data Segregation
    PI 데이터: 별도 Kafka 토픽 (pi-*). Non-PI 데이터: 표준 토픽 (telemetry-*). PI Data: Separate Kafka topics (pi-*). Non-PI Data: Standard topics (telemetry-*).
  3. 접근 제어Access Control
    PI 토픽: IAM + SASL/SCRAM + 감사 로깅. Non-PI 토픽: 표준 IAM 인증. PI topics: IAM + SASL/SCRAM + audit logging. Non-PI topics: Standard IAM authentication.
마이그레이션 참고Migration Note 데이터 흐름에서 VIN 위치가 식별되는 한, 마이그레이션은 주요 차단 요소 없이 진행됩니다. 토큰화는 점진적으로 구현할 수 있습니다. As long as VIN locations are identified in data flows, migration proceeds without major blockers. Tokenization can be implemented incrementally.
