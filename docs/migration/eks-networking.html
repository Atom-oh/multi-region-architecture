EKS 네트워킹EKS Networking

On-Prem Flannel/Calico 기반 네트워킹에서 AWS VPC CNI로 전환합니다. Zone-Aware Routing으로 AZ 간 트래픽 비용을 최적화하고, Network Policy로 마이크로서비스 간 통신을 제어합니다.Transition from On-Prem Flannel/Calico-based networking to AWS VPC CNI. Optimize cross-AZ traffic costs with Zone-Aware Routing and control inter-service communication with Network Policy.

Zone-Aware RoutingZone-Aware Routing

AZ 인지 라우팅으로 Cross-AZ 비용 절감Reduce Cross-AZ Costs with Zone-Aware Routing Network

AWS에서 Cross-AZ 트래픽은 GB당 $0.01의 비용이 발생합니다. Zone-Aware Routing을 통해 동일 AZ 내 Pod 간 통신을 우선하여 60-80%의 비용을 절감할 수 있습니다.Cross-AZ traffic on AWS costs $0.01 per GB. Zone-Aware Routing prioritizes communication between Pods in the same AZ, reducing costs by 60-80%.

TopologySpreadConstraints 설정TopologySpreadConstraints Configuration

Pod를 AZ 간 균등하게 분산하여 고가용성과 비용 최적화를 동시에 달성합니다.Distribute Pods evenly across AZs to achieve both high availability and cost optimization.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 6
  template:
    spec:
      topologySpreadConstraints:
        # Hard constraint: Max 1 Pod difference between AZs
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: api-server
          minDomains: 3    # Ensure 3 AZ distribution
        # Soft constraint: Even distribution across nodes
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: ScheduleAnyway
          labelSelector:
            matchLabels:
              app: api-serveryaml

Topology Aware Hints (EndpointSlice)Topology Aware Hints (EndpointSlice)

Service에 Topology Aware Hints를 활성화하면 kube-proxy가 동일 AZ의 엔드포인트를 우선 선택합니다.Enabling Topology Aware Hints on a Service makes kube-proxy prefer endpoints in the same AZ.

apiVersion: v1
kind: Service
metadata:
  name: api-server
  annotations:
    service.kubernetes.io/topology-aware-hints: "auto"
spec:
  selector:
    app: api-server
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIPyaml

VPC CNIVPC CNI

AWS VPC CNI - Pod에 VPC IP 직접 할당AWS VPC CNI - Direct VPC IP Assignment to Pods Network

On-Prem에서는 Flannel/Calico의 overlay 네트워크를 사용했지만, EKS에서는 VPC CNI가 Pod에 VPC IP를 직접 할당합니다. 이로 인해 encapsulation 오버헤드가 없고 AWS 서비스와의 직접 통신이 가능합니다.While On-Prem used Flannel/Calico overlay networks, EKS VPC CNI assigns VPC IPs directly to Pods. This eliminates encapsulation overhead and enables direct communication with AWS services.

Prefix Delegation으로 IP 효율 향상Improve IP Efficiency with Prefix Delegation

기본적으로 ENI당 Secondary IP 수가 제한되지만, Prefix Delegation을 사용하면 /28 prefix를 할당받아 IP 용량이 16배 증가합니다.By default, secondary IPs per ENI are limited, but Prefix Delegation assigns /28 prefixes, increasing IP capacity 16x.

# Enable Prefix Delegation on aws-node DaemonSet
kubectl set env daemonset aws-node -n kube-system \
  ENABLE_PREFIX_DELEGATION=true \
  WARM_PREFIX_TARGET=1bash

Secondary CIDR 추가Add Secondary CIDR

VPC IP가 부족할 경우 100.64.0.0/10 또는 198.19.0.0/16 대역의 Secondary CIDR을 추가할 수 있습니다.When VPC IPs are exhausted, add Secondary CIDRs from 100.64.0.0/10 or 198.19.0.0/16 ranges.

# Add secondary CIDR to VPC
aws ec2 associate-vpc-cidr-block \
  --vpc-id vpc-12345678 \
  --cidr-block 100.64.0.0/16

# Create subnets in the new CIDR for Pods
aws ec2 create-subnet \
  --vpc-id vpc-12345678 \
  --cidr-block 100.64.0.0/19 \
  --availability-zone us-east-1abash
VPC CNI vs Cilium CNI 선택 기준VPC CNI vs Cilium CNI Selection Criteria VPC CNI: AWS 네이티브 통합이 중요하거나 Security Group for Pods가 필요한 경우 선택합니다. Cilium: eBPF 기반 고급 네트워크 정책, Service Mesh 통합, 멀티 클라우드 이식성이 필요한 경우 선택합니다. 대부분의 EKS 워크로드에서는 VPC CNI가 권장됩니다.VPC CNI: Choose when AWS native integration or Security Group for Pods is required. Cilium: Choose when eBPF-based advanced network policies, Service Mesh integration, or multi-cloud portability is needed. VPC CNI is recommended for most EKS workloads.

Network PolicyNetwork Policy

K8s Network Policy로 마이크로서비스 간 통신 제어Control Microservice Communication with K8s Network Policy Network

On-Prem에서 iptables 규칙으로 관리하던 네트워크 접근 제어를 Kubernetes Network Policy로 전환합니다. 기본 정책으로 모든 트래픽을 차단하고, 필요한 통신만 명시적으로 허용하는 Zero Trust 모델을 적용합니다.Migrate network access control from On-Prem iptables rules to Kubernetes Network Policy. Apply a Zero Trust model with default deny-all policies and explicit allowlists for required communication.

Default Deny-All 정책Default Deny-All Policy

네임스페이스에 기본 차단 정책을 적용하여 명시적으로 허용되지 않은 모든 트래픽을 차단합니다.Apply default deny policy to namespace to block all traffic not explicitly allowed.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}  # Apply to all Pods in namespace
  policyTypes:
    - Ingress
    - Egress
  # No ingress/egress rules = deny allyaml

Namespace 격리 정책Namespace Isolation Policy

같은 네임스페이스 내 Pod 간 통신만 허용하고, 외부 네임스페이스로부터의 접근을 차단합니다.Allow communication only between Pods in the same namespace, blocking access from external namespaces.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-same-namespace
  namespace: production
spec:
  podSelector: {}
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector: {}  # Same namespace onlyyaml

특정 서비스 간 통신 허용Allow Specific Service Communication

API Gateway에서 백엔드 서비스로의 통신만 명시적으로 허용합니다.Explicitly allow communication from API Gateway to backend services.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: product-catalog
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: api-gateway
      ports:
        - protocol: TCP
          port: 8080yaml

외부 Egress 허용 (DNS, AWS 서비스)Allow External Egress (DNS, AWS Services)

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-external-egress
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api-server
  policyTypes:
    - Egress
  egress:
    # Allow DNS
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
    # Allow HTTPS to AWS services
    - to:
        - ipBlock:
            cidr: 0.0.0.0/0
      ports:
        - protocol: TCP
          port: 443yaml
VPC CNI Network Policy 활성화Enable VPC CNI Network Policy VPC CNI에서 Network Policy를 사용하려면 aws-node DaemonSet에 ENABLE_NETWORK_POLICY=true를 설정해야 합니다. 기본적으로 비활성화되어 있으며, 활성화 시 약간의 성능 오버헤드가 발생합니다.To use Network Policy with VPC CNI, set ENABLE_NETWORK_POLICY=true on the aws-node DaemonSet. It is disabled by default, and enabling it incurs slight performance overhead.
