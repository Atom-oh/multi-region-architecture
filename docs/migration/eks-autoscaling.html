오토스케일링 (KEDA + Karpenter + Auto Mode)Autoscaling (KEDA + Karpenter + Auto Mode)

On-Prem Cluster Autoscaler에서 EKS의 다층 오토스케일링으로 전환합니다. Pod 레벨(KEDA/HPA)과 Node 레벨(Karpenter/Auto Mode)을 조합하여 비용 최적화와 빠른 스케일링을 동시에 달성합니다.Transition from On-Prem Cluster Autoscaler to EKS multi-layer autoscaling. Combine Pod-level (KEDA/HPA) and Node-level (Karpenter/Auto Mode) scaling to achieve both cost optimization and fast scaling.

다층 스케일링 조합Multi-Layer Scaling Combination KEDA는 Pod 수를 조절하고, Karpenter/Auto Mode는 Node를 조절합니다. 두 계층을 함께 사용하면 Pod가 증가할 때 Node도 자동으로 확장됩니다.KEDA adjusts Pod count, while Karpenter/Auto Mode adjusts Nodes. Using both layers together ensures Nodes automatically scale when Pods increase.

KEDA (Event-Driven Autoscaling)KEDA (Event-Driven Autoscaling)

KEDA 기반 이벤트 드리븐 오토스케일링KEDA Event-Driven Autoscaling Operations

KEDA(Kubernetes Event-driven Autoscaling)는 50개 이상의 이벤트 소스를 지원하는 오토스케일러입니다. 기본 HPA가 CPU/Memory만 지원하는 반면, KEDA는 Kafka lag, SQS 큐 깊이, Prometheus 메트릭 등 다양한 트리거를 제공합니다.KEDA (Kubernetes Event-driven Autoscaling) is an autoscaler supporting 50+ event sources. While native HPA only supports CPU/Memory, KEDA provides various triggers like Kafka lag, SQS queue depth, and Prometheus metrics.

MSK Consumer Lag 기반 스케일링MSK Consumer Lag Based Scaling

CCDI 파이프라인처럼 Kafka 메시지를 처리하는 워크로드는 consumer lag을 기준으로 스케일링합니다.Workloads processing Kafka messages like CCDI pipeline scale based on consumer lag.

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: ccdi-processor-scaler
  namespace: ccdi
spec:
  scaleTargetRef:
    name: ccdi-processor
  minReplicaCount: 3
  maxReplicaCount: 30
  cooldownPeriod: 300        # 5 min cooldown
  pollingInterval: 30       # check every 30 sec
  triggers:
    - type: kafka
      metadata:
        bootstrapServers: "b-1.productionmskuseast1.qtmqnz.c17.kafka.us-east-1.amazonaws.com:9096"
        consumerGroup: ccdi-processor-group
        topic: vss-data
        lagThreshold: "1000"   # scale out when lag exceeds 1000
        sasl: scram_sha512
        tls: enableyaml

Cron 스케일러 (출퇴근 시간 사전 스케일업)Cron Scaler (Pre-scale for Peak Hours)

예측 가능한 트래픽 패턴에는 Cron 스케일러로 사전 스케일업하여 콜드 스타트를 방지합니다.For predictable traffic patterns, use Cron scaler to pre-scale and prevent cold starts.

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: api-gateway-cron-scaler
spec:
  scaleTargetRef:
    name: api-gateway
  minReplicaCount: 2
  maxReplicaCount: 20
  triggers:
    # 출근 시간대 사전 스케일업
    - type: cron
      metadata:
        timezone: Asia/Seoul
        start: 0 8 * * 1-5       # 평일 08:00
        end: 0 10 * * 1-5        # 평일 10:00
        desiredReplicas: "10"
    # 점심 시간대
    - type: cron
      metadata:
        timezone: Asia/Seoul
        start: 0 11 * * 1-5
        end: 0 14 * * 1-5
        desiredReplicas: "15"yaml

Prometheus 메트릭 스케일러Prometheus Metric Scaler

RPS(Requests Per Second)나 커스텀 비즈니스 메트릭을 기반으로 스케일링합니다.Scale based on RPS (Requests Per Second) or custom business metrics.

triggers:
  - type: prometheus
    metadata:
      serverAddress: http://prometheus.monitoring.svc:9090
      metricName: http_requests_per_second
      threshold: "100"
      query: sum(rate(http_requests_total{app="api-gateway"}[2m]))yaml
Consumer Group Rebalancing 스케일링 시 consumer rebalancing을 최소화하려면 partition.assignment.strategy=cooperative-sticky를 사용하세요. Cooperative rebalancing은 전체 중단 없이 점진적으로 파티션을 재할당합니다.To minimize consumer rebalancing during scaling, use partition.assignment.strategy=cooperative-sticky. Cooperative rebalancing reassigns partitions incrementally without full stop.

KarpenterKarpenter

Karpenter 노드 오토스케일링Karpenter Node Autoscaling Infrastructure

Karpenter는 Cluster Autoscaler를 대체하는 차세대 노드 오토스케일러입니다. Node Group 없이 워크로드 요구사항에 맞는 최적의 인스턴스를 초 단위로 프로비저닝합니다.Karpenter is a next-generation node autoscaler replacing Cluster Autoscaler. It provisions optimal instances matching workload requirements in seconds without Node Groups.

Spot 인터럽트 처리Spot Interrupt Handling

Spot 인스턴스 중단 시 Karpenter가 자동으로 대체 노드를 프로비저닝합니다. 중요한 워크로드는 annotation으로 보호합니다.When Spot instances are interrupted, Karpenter automatically provisions replacement nodes. Protect critical workloads with annotations.

metadata:
  annotations:
    karpenter.sh/do-not-disrupt: "true"  # 이 Pod가 있는 노드는 중단하지 않음yaml

Consolidation 정책Consolidation Policy

사용률이 낮은 노드를 자동으로 정리하여 비용을 최적화합니다.Automatically clean up underutilized nodes to optimize costs.

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: general
spec:
  disruption:
    # WhenEmpty: 빈 노드만 제거
    # WhenEmptyOrUnderutilized: 사용률 낮은 노드도 제거 (비용 최적화)
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 30s
    expireAfter: 720h  # 30일 후 노드 교체 (AMI 최신화)yaml
Karpenter vs Cluster AutoscalerKarpenter vs Cluster Autoscaler Cluster Autoscaler는 Node Group 단위로 스케일링하여 분 단위가 걸리지만, Karpenter는 Pod 요구사항을 직접 분석하여 최적의 인스턴스를 초 단위로 프로비저닝합니다. 또한 다양한 인스턴스 타입을 동적으로 선택하여 Spot 용량 확보에 유리합니다.Cluster Autoscaler scales at Node Group level taking minutes, while Karpenter directly analyzes Pod requirements and provisions optimal instances in seconds. It also dynamically selects various instance types, improving Spot capacity availability.

EKS Auto ModeEKS Auto Mode

EKS Auto Mode - AWS 완전 관리형 노드EKS Auto Mode - Fully AWS Managed Nodes Infrastructure

EKS Auto Mode는 AWS가 Karpenter를 내부적으로 운영하며, 노드 관리를 완전히 자동화합니다. 사용자는 Karpenter를 직접 설치/관리할 필요 없이 Auto Mode를 활성화하기만 하면 됩니다.EKS Auto Mode runs Karpenter internally by AWS, fully automating node management. Users only need to enable Auto Mode without installing/managing Karpenter directly.

기존 Karpenter에서 Auto Mode 전환 고려사항Considerations for Migrating from Karpenter to Auto Mode

  • 점진적 전환: 기존 Karpenter NodePool과 Auto Mode NodePool을 병행 운영 후 점진적으로 마이그레이션Gradual migration: Run existing Karpenter NodePools and Auto Mode NodePools in parallel, then migrate gradually
  • NodePool 매핑: 기존 NodePool 설정을 Auto Mode의 NodePool로 재구성NodePool mapping: Reconfigure existing NodePool settings to Auto Mode NodePools
  • 테스트 환경 우선: 개발/스테이징 환경에서 먼저 검증 후 프로덕션 적용Test first: Validate in dev/staging environments before production

Drift 감지 및 자동 교체Drift Detection and Auto Replacement

NodePool 설정이 변경되면 Auto Mode가 자동으로 기존 노드를 새 설정으로 교체합니다.When NodePool settings change, Auto Mode automatically replaces existing nodes with new settings.

# Drift가 감지되는 변경 사항
- NodePool requirements 변경      # Drift 발생
- NodeClass AMI family 변경       # Drift 발생
- NodeClass block device 변경    # Drift 발생

# Drift가 감지되지 않는 변경 사항
- NodePool weight 변경           # Drift 미발생
- NodePool limits 변경           # Drift 미발생text
Auto Mode 제약 사항Auto Mode Limitations Auto Mode는 Custom AMI를 사용할 수 없으며, 지원되는 AMI Family는 AL2023과 Bottlerocket입니다. GPU NodePool은 일부 제한이 있으므로 GPU 워크로드가 많은 경우 직접 Karpenter 운영을 고려하세요. Windows 노드는 지원되지 않습니다.Auto Mode cannot use Custom AMIs, and supported AMI families are AL2023 and Bottlerocket. GPU NodePools have some limitations, so consider direct Karpenter operation for GPU-heavy workloads. Windows nodes are not supported.
