6개 NodePool 전략6 NodePool Strategy

Karpenter v1.9 - 워크로드별 NodePool 분리Karpenter v1.9 - Workload-specific NodePool Isolation Infrastructure

Karpenter v1.9는 워크로드 특성에 맞는 인스턴스를 자동으로 선택하고 프로비저닝합니다. 6개의 NodePool로 워크로드를 분리하여 비용 최적화와 성능을 동시에 달성합니다.Karpenter v1.9 automatically selects and provisions instances matching workload characteristics. Workloads are separated into 6 NodePools to achieve both cost optimization and performance.

General NodePool 정의General NodePool Definition

k8s/infra/karpenter/general-nodepool.yaml

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: general
spec:
  template:
    metadata:
      labels:
        node-pool: general
    spec:
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: [spot, on-demand]
        - key: kubernetes.io/arch
          operator: In
          values: [amd64]
        - key: karpenter.k8s.aws/instance-family
          operator: In
          values: [m6i, m7i, c6i, c7i, r6i, r7i]
        - key: karpenter.k8s.aws/instance-size
          operator: In
          values: [large, xlarge, 2xlarge]
  weight: 70
  limits:
    cpu: "200"
    memory: 400Gi
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 30syaml
Spot 전략Spot Strategy 현재는 amd64 인스턴스만 사용하지만, arm64(Graviton) 추가 시 비용 최적화가 가능합니다. Spot 인스턴스 중단 시 Karpenter가 자동으로 대체 노드를 프로비저닝하며, consolidateAfter: 30s로 빈 노드를 빠르게 정리합니다.Currently using amd64 instances only, but adding arm64 (Graviton) can further optimize costs. When Spot instances are interrupted, Karpenter automatically provisions replacement nodes, and consolidateAfter: 30s quickly cleans up empty nodes.

EC2NodeClass

EC2NodeClass - AMI, 스토리지, 네트워크 구성EC2NodeClass - AMI, Storage, Network Configuration Infrastructure

EC2NodeClass는 Karpenter가 프로비저닝하는 EC2 인스턴스의 AMI, 스토리지, 네트워크 설정을 정의합니다. 모든 NodePool이 동일한 default EC2NodeClass를 참조합니다.EC2NodeClass defines the AMI, storage, and network settings for EC2 instances provisioned by Karpenter. All NodePools reference the same default EC2NodeClass.

k8s/infra/karpenter/ec2nodeclass.yaml

apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
metadata:
  name: default
spec:
  amiSelectorTerms:
    - alias: al2023@latest      # Amazon Linux 2023 latest
  blockDeviceMappings:
    - deviceName: /dev/xvda
      ebs:
        volumeSize: 100Gi
        volumeType: gp3
        iops: 3000
        throughput: 125
        encrypted: true
        deleteOnTermination: true
  role: multi-region-mall-node-group   # IAM role (global, same for both regions)
  subnetSelectorTerms:
    - tags:
        Tier: private              # Use private subnets only
  securityGroupSelectorTerms:
    - tags:
        karpenter.sh/discovery: multi-region-mall
  tags:
    Environment: production
    ManagedBy: karpenteryaml

DSQL Active-Active

DSQL - 네이티브 Active-Active 구성DSQL - Native Active-Active Configuration Database

DSQL은 서버리스 분산 SQL (PostgreSQL 16 호환)로, 별도의 복제 설정 없이 AWS 네이티브로 멀티리전 Active-Active(양 리전에서 동시 읽기/쓰기 가능한 구성)를 지원합니다. 각 리전에 DSQL 클러스터를 생성하고 multi-region-properties로 피어링을 설정하면, AWS가 Peer Linking을 통해 데이터를 자동 동기화합니다.DSQL is a serverless distributed SQL (PostgreSQL 16 compatible) that natively supports multi-region Active-Active (both regions handle reads and writes simultaneously) without separate replication setup. After creating DSQL clusters in each region and configuring peering via multi-region-properties, AWS automatically synchronizes data through Peer Linking.

Terraform 정의Terraform Definition

terraform/modules/data/dsql/main.tf

resource "aws_dsql_cluster" "this" {
  deletion_protection_enabled = var.deletion_protection_enabled

  tags = merge(var.tags, {
    Name        = "${var.environment}-dsql-${var.region}"
    Environment = var.environment
  })
}hcl
DSQL Active-Active의 장점Benefits of DSQL Active-Active 기존 Aurora Global Database와 달리, DSQL은 양 리전에서 동시 쓰기가 가능합니다. Write Forwarding이 필요 없어 쓰기 지연(latency)이 크게 줄어듭니다. 멀티리전 구성은 각 리전에 DSQL 클러스터를 생성하고, multi-region-properties로 상호 피어링을 설정하면 AWS가 데이터 복제를 자동 처리합니다. Witness Region(트랜잭션 로그만 저장하는 제3 리전)도 지정해야 합니다.Unlike Aurora Global Database, DSQL allows simultaneous writes in both regions. Without Write Forwarding, write latency is significantly reduced. For multi-region setup, create DSQL clusters in each region and configure mutual peering with multi-region-properties, and AWS handles data replication automatically. A Witness Region (a third region storing only transaction logs) must also be specified.

MSK Replicator

MSK Replicator - 크로스리전 토픽 복제MSK Replicator - Cross-Region Topic Replication Database

MSK Replicator는 AWS 네이티브 크로스리전 Kafka 토픽 복제 서비스입니다. MirrorMaker2를 직접 운영할 필요 없이, 소스/타겟 클러스터를 지정하면 자동으로 토픽과 컨슈머 그룹을 복제합니다.MSK Replicator is an AWS-native cross-region Kafka topic replication service. Without managing MirrorMaker2 directly, it automatically replicates topics and consumer groups when you specify source/target clusters.

Terraform 정의 (발췌)Terraform Definition (excerpt)

terraform/modules/data/msk/main.tf

resource "aws_msk_replicator" "this" {
  count = var.enable_replicator ? 1 : 0

  replicator_name = "${var.environment}-msk-replicator-${var.region}"

  # Source cluster
  kafka_cluster {
    amazon_msk_cluster {
      msk_cluster_arn = var.source_cluster_arn
    }
    vpc_config {
      subnet_ids          = var.data_subnet_ids
      security_groups_ids = [var.security_group_id]
    }
  }

  # Target cluster
  kafka_cluster {
    amazon_msk_cluster {
      msk_cluster_arn = var.target_cluster_arn
    }
    vpc_config {
      subnet_ids          = var.data_subnet_ids
      security_groups_ids = [var.security_group_id]
    }
  }

  replication_info_list {
    source_kafka_cluster_arn = var.source_cluster_arn
    target_kafka_cluster_arn = var.target_cluster_arn
    target_compression_type  = "GZIP"

    topic_replication {
      topics_to_replicate                  = var.replicator_topics
      copy_access_control_lists_for_topics = true
      copy_topic_configurations            = true
      detect_and_copy_new_topics           = true
    }

    consumer_group_replication {
      consumer_groups_to_replicate         = [".*"]
      synchronise_consumer_group_offsets   = true
      detect_and_copy_new_consumer_groups  = true
    }
  }

  service_execution_role_arn = aws_iam_role.msk_replicator[0].arn
}hcl
핵심 설정Key Settings detect_and_copy_new_topics: true로 새 토픽이 자동 복제됩니다. synchronise_consumer_group_offsets: true로 failover 시 컨슈머가 이어서 처리할 수 있습니다. 단, offset 동기화에는 10-30초 지연이 있으므로, failover 시 소수의 메시지가 재처리될 수 있습니다.With detect_and_copy_new_topics: true, new topics are automatically replicated. With synchronise_consumer_group_offsets: true, consumers can resume processing during failover. Note: offset synchronization has a 10-30 second lag, so a small number of messages may be reprocessed during failover.
MSK Replicator 방향성MSK Replicator Directionality MSK Replicator는 단방향(source→target) 복제입니다. 양방향 복제가 필요한 경우 2개의 Replicator를 각각 반대 방향으로 구성해야 합니다.MSK Replicator is unidirectional (source→target) replication. For bidirectional replication, configure 2 Replicators in opposite directions.

OpenSearch CCR 제약사항OpenSearch CCR Limitations CCR은 leader→follower 단방향 복제이며, follower 인덱스는 읽기 전용입니다. 사전 조건으로 Fine-Grained Access Control과 Node-to-node encryption이 활성화되어 있어야 합니다.CCR is unidirectional leader→follower replication, and follower indices are read-only. Prerequisites: Fine-Grained Access Control and Node-to-node encryption must be enabled.
ElastiCache 주의ElastiCache Caution ElastiCache(Valkey)의 Global Datastore는 리전간 비동기 복제를 자동으로 수행합니다 (Secondary는 읽기 전용 복제본). 다만, 캐시 특성상 각 리전에서 독립적으로 운영하는 것이 더 효율적이므로, Global Datastore는 주로 DR(재해 복구) 시 Secondary를 Primary로 승격하기 위한 목적으로 구성합니다. cache miss 시 DB에서 재로드합니다. Secondary 클러스터의 engine, engine_version, 암호화 파라미터는 null로 설정해야 합니다 (Global Datastore에서 상속).ElastiCache (Valkey) Global Datastore performs automatic async replication between regions (Secondary is read-only replica). However, due to cache characteristics, independent operation per region is more efficient, so Global Datastore is mainly configured for promoting Secondary to Primary during DR (disaster recovery). Cache misses reload from DB. Secondary cluster's engine, engine_version, and encryption parameters must be set to null (inherited from Global Datastore).
