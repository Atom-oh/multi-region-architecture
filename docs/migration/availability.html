CCS 가용성 아키텍처CCS Availability Architecture

CCS 가용성은 타협할 수 없습니다 — P1 인시던트는 회장에게 직접 보고됩니다. 이 섹션에서는 인시던트 대응, 네트워크 연결, DR/장애조치 전략, MongoDB 마이그레이션 경로를 다룹니다.CCS availability is non-negotiable — P1 incidents report directly to the Chairman. This section covers incident response, network connectivity, DR/failover strategy, and the MongoDB migration path.

P1 인시던트 대응 프레임워크P1 Incident Response Framework

CCS P1 인시던트 대응 프레임워크CCS P1 Incident Response Framework Operations

CCS는 P1 인시던트가 회장에게 직접 보고되는 "일상 CIO" 보고 구조로 운영됩니다. 이로 인해 서비스 중단에 대한 제로 톨러런스가 적용되며 모든 운영 지표에 엄격한 SLA가 요구됩니다.CCS operates under an "Everyday CIO" reporting structure where P1 incidents escalate directly to the Chairman. This creates zero tolerance for service outages and requires stringent SLAs across all operational metrics.

Verizon 프라이빗 네트워크 연결Verizon Private Network Connectivity

Verizon 프라이빗 네트워크에서 AWS 연결Verizon Private Network to AWS Connectivity Network

차량 연결은 Verizon Private IP(PIP) 네트워크를 통해 이루어집니다. 마이그레이션에는 기존 온프레미스 경로를 유지하면서 Verizon에서 AWS로의 보안 연결을 설정해야 합니다.Vehicle connectivity flows through Verizon Private IP (PIP) network. The migration requires establishing secure connectivity from Verizon to AWS while maintaining the existing On-Prem path.

멀티 리전 DR/장애조치 전략Multi-Region DR/Failover Strategy

멀티 리전 DR/장애조치 전략Multi-Region DR/Failover Strategy Network

현재 CCS 상태: DR 기능 없음 — 기존 사고방식은 인시던트 발생 후 "OTA로 대응"하는 것입니다. AWS 마이그레이션은 Route53 헬스 체크를 통한 자동 장애조치가 포함된 Active-Active 아키텍처를 도입합니다.Current CCS status: NO DR capability — the prevailing mentality is "respond via OTA" after incidents occur. The AWS migration introduces Active-Active architecture with automatic failover via Route53 health checks.

Route53 헬스 체크 + 지연시간 기반 라우팅Route53 Health Check + Latency Routing

terraform/modules/edge/route53/main.tf

# Health Check (per region)
resource "aws_route53_health_check" "regional" {
  for_each = var.lb_dns_names

  fqdn              = each.value
  port              = 80
  type              = "HTTP"
  resource_path     = var.health_check_path
  request_interval  = var.health_check_interval
  failure_threshold = var.health_check_failure_threshold

  tags = merge(var.tags, {
    Name   = "${var.environment}-${each.key}-health-check"
    Region = each.key
  })
}

# Latency-based Routing (per region)
resource "aws_route53_record" "api_latency" {
  for_each = var.lb_dns_names

  zone_id = var.zone_id
  name    = "api-internal.${data.aws_route53_zone.main.name}"
  type    = "A"

  alias {
    name                   = each.value
    zone_id                = var.lb_zone_ids[each.key]
    evaluate_target_health = true
  }

  set_identifier = each.key

  latency_routing_policy {
    region = each.key
  }

  health_check_id = aws_route53_health_check.regional[each.key].id
}

# CloudWatch Alarm for Health Check monitoring
resource "aws_cloudwatch_metric_alarm" "health_check" {
  for_each = var.lb_dns_names

  alarm_name          = "${var.environment}-${each.key}-health-check-alarm"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 2
  metric_name         = "HealthCheckStatus"
  namespace           = "AWS/Route53"
  period              = 60
  threshold           = 1
  treat_missing_data  = "breaching"
}hcl
