EKS 클러스터 업그레이드 전략EKS Cluster Upgrade Strategy

EKS 멀티 리전 클러스터 업그레이드 전략EKS Multi-Region Cluster Upgrade Strategy Infrastructure

멀티 리전 EKS 클러스터는 반드시 동일한 버전을 유지해야 합니다. 현재 두 리전 모두 EKS 1.35를 운영 중이며, 순차적 업그레이드 전략으로 다운타임 없이 버전을 관리합니다.Multi-region EKS clusters must maintain the same version. Both regions currently run EKS 1.35, with sequential upgrade strategy managing versions without downtime.

순차적 업그레이드 절차Sequential Upgrade Procedure

  1. Secondary (us-west-2) 먼저 업그레이드Upgrade Secondary (us-west-2) first — 트래픽 비중이 낮은 리전에서 먼저 검증Validate in lower-traffic region first
  2. 24시간 안정화 관찰24-hour stabilization observation — 메트릭, 로그, 알람 모니터링Monitor metrics, logs, alarms
  3. Primary (us-east-1) 업그레이드Upgrade Primary (us-east-1) — Secondary 검증 완료 후 진행Proceed after Secondary validation
  4. Addon 버전 동기화Addon version sync — Control plane 업그레이드 후 addon도 함께 업그레이드Upgrade addons after control plane upgrade

IRSA 필수IRSA Required EKS 1.35에서는 IMDS 접근이 제한되어 EBS/EFS CSI 드라이버에 IRSA(IAM Roles for Service Accounts)가 필수입니다. Pod에서 EC2 메타데이터 서비스에 직접 접근할 수 없으므로, CSI 드라이버가 AWS API를 호출하려면 반드시 Service Account에 IAM Role을 연결해야 합니다.In EKS 1.35, IMDS access is restricted, requiring IRSA (IAM Roles for Service Accounts) for EBS/EFS CSI drivers. Since pods cannot directly access EC2 metadata service, CSI drivers must have IAM Roles attached to Service Accounts to call AWS APIs.

CCDI 워크로드 오토스케일링CCDI Workload Autoscaling

KEDA 기반 CCDI 워크로드 오토스케일링CCDI Workload Autoscaling with KEDA Operations

현재 CCDI의 Pod 수는 Kafka 파티션 수와 1:1로 고정되어 있어 HPA가 적용되지 않습니다. 피크 시간대(10AM-1PM)에는 메모리 사용률이 90-100%에 달하지만, 그 외 시간에는 20% 미만입니다.Currently, CCDI pod count is fixed 1:1 with Kafka partition count, so HPA is not applied. Memory usage reaches 90-100% during peak hours (10AM-1PM), but stays below 20% at other times.

현재 문제점Current Issues

  • Pod 수 = Kafka 파티션 수 (고정) — 탄력적 스케일링 불가Pod count = Kafka partition count (fixed) - no elastic scaling
  • 피크 시간대 메모리 부족 위험Memory shortage risk during peak hours
  • 비피크 시간대 리소스 낭비Resource waste during off-peak hours

KEDA ScaledObject 제안KEDA ScaledObject Proposal

Kafka consumer lag 기반 스케일링으로 워크로드에 맞는 탄력적 확장/축소를 구현합니다.Implements elastic scaling based on Kafka consumer lag to match workload demands.

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: ccdi-processor-scaler
  namespace: ccdi
spec:
  scaleTargetRef:
    name: ccdi-processor
  minReplicaCount: 3
  maxReplicaCount: 30
  cooldownPeriod: 300        # 5 min cooldown
  pollingInterval: 30       # check every 30 sec
  triggers:
    - type: kafka
      metadata:
        bootstrapServers: "b-1.productionmskuseast1.qtmqnz.c17.kafka.us-east-1.amazonaws.com:9096"
        consumerGroup: ccdi-processor-group
        topic: vss-data
        lagThreshold: "1000"   # scale out when lag exceeds 1000
        sasl: scram_sha512
        tls: enableyaml

스케일링 파라미터Scaling Parameters

파라미터Parameter값Value설명Description
minReplicaCount3최소 Pod 수 (3 AZ 분산)Min pods (distributed across 3 AZs)
maxReplicaCount30최대 Pod 수 (파티션 수 기준)Max pods (based on partition count)
cooldownPeriod300s스케일 다운 대기 시간Scale down wait time
lagThreshold1000스케일 트리거 lag 임계값Scale trigger lag threshold
Consumer Group Rebalancing 스케일링 시 consumer rebalancing을 최소화하려면 partition.assignment.strategy=cooperative-sticky를 사용하세요. Cooperative rebalancing은 전체 중단 없이 점진적으로 파티션을 재할당합니다.To minimize consumer rebalancing during scaling, use partition.assignment.strategy=cooperative-sticky. Cooperative rebalancing reassigns partitions incrementally without full stop.

Karpenter 6개 NodePoolKarpenter 6 NodePools

Karpenter 6개 NodePool - 멀티 리전 고려사항Karpenter 6 NodePools - Multi-Region Considerations Infrastructure

Karpenter v1.9는 6개의 NodePool로 워크로드를 분리하여 비용 최적화와 성능을 동시에 달성합니다. 리전별로 인스턴스 가용성과 AZ 매핑이 다를 수 있으므로 Spot 전략에 유연성이 필요합니다.Karpenter v1.9 separates workloads into 6 NodePools to achieve both cost optimization and performance. Spot strategy needs flexibility as instance availability and AZ mapping may differ by region.

Spot 전략Spot Strategy 현재는 amd64 인스턴스만 사용하지만, arm64(Graviton) 추가 시 비용 최적화가 가능합니다. Spot 인스턴스 중단 시 Karpenter가 자동으로 대체 노드를 프로비저닝하며, consolidateAfter: 30s로 빈 노드를 빠르게 정리합니다.Currently using amd64 instances only, but adding arm64 (Graviton) can further optimize costs. When Spot instances are interrupted, Karpenter automatically provisions replacement nodes, and consolidateAfter: 30s quickly cleans up empty nodes.

ArgoCD 멀티 클러스터 GitOpsArgoCD Multi-Cluster GitOps

ArgoCD 멀티 클러스터 GitOps 배포ArgoCD Multi-Cluster GitOps Deployment GitOps

ArgoCD ApplicationSet의 clusters generator를 사용하여 리전별 Application을 자동 생성합니다. Kustomize overlay로 리전별 엔드포인트(DSQL, MongoDB Atlas, MSK, OpenSearch)를 주입합니다.Uses ArgoCD ApplicationSet's clusters generator to automatically create per-region Applications. Injects region-specific endpoints (DSQL, MongoDB Atlas, MSK, OpenSearch) via Kustomize overlays.

ApplicationSet - Core ServicesApplicationSet - Core Services

k8s/infra/argocd/apps/appset-core.yaml

apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: core-services
  namespace: argocd
spec:
  generators:
    - clusters:
        selector:
          matchExpressions:
            - key: region
              operator: Exists
  template:
    metadata:
      name: 'core-{{metadata.labels.region}}'
    spec:
      project: default
      source:
        repoURL: git@github.com:Atom-oh/multi-region-architecture.git
        targetRevision: main
        path: 'k8s/overlays/{{metadata.labels.region}}/core'
      destination:
        server: '{{server}}'
        namespace: core-services
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
        retry:
          limit: 5
          backoff:
            duration: 5s
            factor: 2
            maxDuration: 3myaml
