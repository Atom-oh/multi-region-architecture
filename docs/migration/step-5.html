Step 5: AWS 트래픽 확대Traffic Scale-Up (5:2.5:2.5)

AWS로의 트래픽 비중을 50%로 확대하면서 Observability 스택을 통해 서비스 안정성을 검증합니다. OTel (OpenTelemetry) Collector는 Step 1부터 배포되어 있지만, 트래픽이 50%로 확대되는 이 시점에서 sampling 전략과 모니터링 임계값 튜닝이 중요해지므로 여기서 상세히 다룹니다.Scale up traffic to AWS to 50% while validating service stability through the Observability stack. OTel (OpenTelemetry) Collector has been deployed since Step 1, but sampling strategy and monitoring threshold tuning become critical at this stage when traffic reaches 50%, so we cover it in detail here.

OTel (OpenTelemetry) Collector Pipeline

Tail-based Sampling + 듀얼 Export (Tempo + X-Ray)Tail-based Sampling + Dual Export (Tempo + X-Ray) Observability

OTel Collector는 DaemonSet으로 모든 노드에 배포되며, 수집한 트레이스를 Grafana Tempo와 AWS X-Ray에 동시에 전송합니다. Tail-based sampling으로 중요한 트레이스만 선별하여 비용을 절감합니다.OTel Collector is deployed as a DaemonSet on all nodes, sending collected traces to both Grafana Tempo and AWS X-Ray simultaneously. Tail-based sampling selects only important traces to reduce costs.

Sampling 정책Sampling Policy

k8s/infra/otel-collector/otel-collector.yaml (발췌)

tail_sampling:
  decision_wait: 10s
  num_traces: 100000
  expected_new_traces_per_sec: 1000
  policies:
    - name: errors-policy
      type: status_code
      status_code:
        status_codes:
          - ERROR              # Error traces: 100% collection
    - name: slow-requests-policy
      type: latency
      latency:
        threshold_ms: 500    # Over 500ms: 100% collection
    - name: probabilistic-policy
      type: probabilistic
      probabilistic:
        sampling_percentage: 10  # Default: 10% samplingyaml

Tail-based vs Head-based Head-based sampling은 트레이스 시작 시점에 결정하므로, 나중에 에러가 발생해도 놓칠 수 있습니다. Tail-based는 전체 트레이스가 완료된 후 판단하므로, 에러/슬로우 트레이스를 100% 캡처합니다. 단, 메모리 사용량이 높아 num_traces: 100000으로 버퍼를 충분히 잡아야 합니다.Head-based sampling decides at trace start, so errors occurring later may be missed. Tail-based decides after the entire trace completes, capturing 100% of error/slow traces. However, memory usage is high, so set num_traces: 100000 for sufficient buffer.

Grafana Tempo + CloudWatch

Observability 스택 구성과 모니터링 대시보드Observability Stack & Monitoring Dashboard Observability

Observability 스택은 3개 계층으로 구성됩니다: Metrics (CloudWatch + Prometheus), Logs (CloudWatch Logs), Traces (Tempo + X-Ray). Grafana에서 3가지 신호를 통합하여 크로스리전 서비스 상태를 모니터링합니다.The Observability stack consists of 3 layers: Metrics (CloudWatch + Prometheus), Logs (CloudWatch Logs), Traces (Tempo + X-Ray). Grafana integrates all 3 signals to monitor cross-region service status.
