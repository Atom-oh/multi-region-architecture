Step 3: 리전간 동기화 활성화Cross-Region Sync Activation

us-west-2와 us-east-1 간의 데이터 동기화를 활성화합니다. DSQL Active-Active, DocumentDB Global Cluster, MSK Replicator, OpenSearch CCR (Cross-Cluster Replication)이 활성화됩니다.

트래픽 분배Traffic Distribution On-Prem 90% / us-west-2 10% / us-east-1 0% — 동기화 안정성 확인 후 트래픽 전환

DSQL Active-Active

Aurora DSQL - 네이티브 Active-Active 구성Aurora DSQL - Native Active-Active Configuration Database

Aurora DSQL은 서버리스 분산 SQL (PostgreSQL 16 호환)로, 별도의 복제 설정 없이 AWS 네이티브로 멀티리전 Active-Active를 지원합니다. 각 리전에 독립적인 DSQL 클러스터를 생성하면, AWS가 자동으로 Peer Linking을 통해 데이터를 동기화합니다.

Terraform 정의

terraform/modules/data/dsql/main.tf

resource "aws_dsql_cluster" "this" {
  deletion_protection_enabled = var.deletion_protection_enabled

  tags = merge(var.tags, {
    Name        = "${var.environment}-dsql-${var.region}"
    Environment = var.environment
  })
}hcl
DSQL Active-Active의 장점 기존 Aurora Global Database와 달리, DSQL은 양 리전에서 동시 쓰기가 가능합니다. Write Forwarding이 필요 없어 쓰기 지연(latency)이 크게 줄어듭니다. 멀티리전 구성은 각 리전에 DSQL 클러스터를 생성하고, multi-region-properties로 상호 피어링을 설정하면 AWS가 데이터 복제를 자동 처리합니다. Witness Region(트랜잭션 로그만 저장하는 제3 리전)도 지정해야 합니다.

MSK Replicator

MSK Replicator - 크로스리전 토픽 복제MSK Replicator - Cross-Region Topic Replication Database

MSK Replicator는 AWS 네이티브 크로스리전 Kafka 토픽 복제 서비스입니다. MirrorMaker2를 직접 운영할 필요 없이, 소스/타겟 클러스터를 지정하면 자동으로 토픽과 컨슈머 그룹을 복제합니다.

Terraform 정의 (발췌)

terraform/modules/data/msk/main.tf

resource "aws_msk_replicator" "this" {
  count = var.enable_replicator ? 1 : 0

  replicator_name = "${var.environment}-msk-replicator-${var.region}"

  # Source cluster
  kafka_cluster {
    amazon_msk_cluster {
      msk_cluster_arn = var.source_cluster_arn
    }
    vpc_config {
      subnet_ids          = var.data_subnet_ids
      security_groups_ids = [var.security_group_id]
    }
  }

  # Target cluster
  kafka_cluster {
    amazon_msk_cluster {
      msk_cluster_arn = var.target_cluster_arn
    }
    vpc_config {
      subnet_ids          = var.data_subnet_ids
      security_groups_ids = [var.security_group_id]
    }
  }

  replication_info_list {
    source_kafka_cluster_arn = var.source_cluster_arn
    target_kafka_cluster_arn = var.target_cluster_arn
    target_compression_type  = "GZIP"

    topic_replication {
      topics_to_replicate                  = var.replicator_topics
      copy_access_control_lists_for_topics = true
      copy_topic_configurations            = true
      detect_and_copy_new_topics           = true
    }

    consumer_group_replication {
      consumer_groups_to_replicate         = [".*"]
      synchronise_consumer_group_offsets   = true
      detect_and_copy_new_consumer_groups  = true
    }
  }

  service_execution_role_arn = aws_iam_role.msk_replicator[0].arn
}hcl
핵심 설정Key Settings detect_and_copy_new_topics: true로 새 토픽이 자동 복제됩니다. synchronise_consumer_group_offsets: true로 failover 시 컨슈머가 이어서 처리할 수 있습니다.
MSK Replicator 방향성 MSK Replicator는 단방향(source→target) 복제입니다. 양방향 복제가 필요한 경우 2개의 Replicator를 각각 반대 방향으로 구성해야 합니다.

DocumentDB + ElastiCache + OpenSearch

Global Cluster, Global Datastore, Cross-Cluster Replication Database

나머지 3개 데이터스토어는 각각의 AWS 네이티브 크로스리전 복제 메커니즘을 사용합니다.

DocumentDB Global Cluster

terraform/modules/data/documentdb-global/main.tf

resource "aws_docdb_cluster" "this" {
  cluster_identifier        = local.cluster_identifier
  # Primary: null (standalone) / Secondary: global cluster ID
  global_cluster_identifier = var.is_primary ? null : var.global_cluster_identifier

  engine         = "docdb"
  engine_version = "8.0.0"

  # Primary needs credentials, secondary inherits
  master_username = var.is_primary ? "docdb_admin" : null
  master_password = var.is_primary ? var.master_password : null
}hcl

ElastiCache Global Datastore

terraform/modules/data/elasticache-global/main.tf

resource "aws_elasticache_global_replication_group" "this" {
  count = var.is_primary ? 1 : 0

  global_replication_group_id_suffix = "${var.environment}-global"
  primary_replication_group_id       = aws_elasticache_replication_group.this.id

  global_replication_group_description = "Global replication group for ${var.environment}"
}hcl

리전간 동기화 요약

서비스동기화 방식방향지연
DocumentDBGlobal ClusterPrimary → Secondary~1s
ElastiCacheGlobal DatastoreGlobal Datastore (DR용 비동기 복제)N/A
OpenSearchCross-Cluster ReplicationLeader → Follower~seconds
OpenSearch CCR 제약사항 CCR은 leader→follower 단방향 복제이며, follower 인덱스는 읽기 전용입니다. 사전 조건으로 Fine-Grained Access Control과 Node-to-node encryption이 활성화되어 있어야 합니다.
ElastiCache 주의 ElastiCache(Valkey)의 Global Datastore는 리전간 비동기 복제를 자동으로 수행합니다 (Secondary는 읽기 전용 복제본). 다만, 캐시 특성상 각 리전에서 독립적으로 운영하는 것이 더 효율적이므로, Global Datastore는 주로 DR(재해 복구) 시 Secondary를 Primary로 승격하기 위한 목적으로 구성합니다. cache miss 시 DB에서 재로드합니다. Secondary 클러스터의 engine, engine_version, 암호화 파라미터는 null로 설정해야 합니다 (Global Datastore에서 상속).
