Step 1: us-west-2 배포 + 듀얼 오리진Deploy + Dual Origin (9:1)

us-west-2 리전에 전체 인프라를 배포하고, Route53 weighted routing으로 On-Prem 90%, us-west-2 10%의 트래픽 비율을 설정합니다. (CloudFront Origin Group은 failover 전용이므로, 가중치 기반 분산은 Route53에서 처리합니다.) EC2 Sync Connector가 On-Prem과 AWS 간 데이터 동기화를 수행합니다. On-Prem DC가 us-west 지역에 위치하므로, 네트워크 지연이 적은 us-west-2를 먼저 배포하여 검증합니다.

트래픽 분배Traffic Distribution On-Prem 90% / us-west-2 10% — Route53 weighted routing으로 소량 트래픽 검증

ArgoCD App of Apps 패턴Pattern

ArgoCD ApplicationSet + Kustomize 오버레이 GitOps

ArgoCD ApplicationSet의 clusters generator를 사용하여, 하나의 ApplicationSet 정의로 양 리전에 동일한 서비스를 자동 배포합니다. 리전별 차이(엔드포인트, SG ID, 인증서 ARN 등)는 Kustomize 오버레이에서 처리합니다.

ApplicationSet: Core 서비스

k8s/infra/argocd/apps/appset-core.yaml

apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: core-services
  namespace: argocd
spec:
  generators:
    - clusters:
        selector:
          matchExpressions:
            - key: region
              operator: Exists
  template:
    metadata:
      name: 'core-{{metadata.labels.region}}'
    spec:
      project: default
      source:
        repoURL: git@github.com:Atom-oh/multi-region-architecture.git
        targetRevision: main
        path: 'k8s/overlays/{{metadata.labels.region}}/core'
      destination:
        server: '{{server}}'
        namespace: core-services
      syncPolicy:
        automated:
          prune: true
          selfHeal: trueyaml

Kustomize: 리전별 ConfigMap 오버레이

k8s/overlays/us-east-1/kustomization.yaml (발췌)

configMapGenerator:
  - name: region-config
    namespace: platform
    literals:
      - REGION=us-east-1
      - REGION_ROLE=PRIMARY
      - DOCUMENTDB_ENDPOINT=production-docdb-global-us-east-1.cluster-c4pe2u8kgt26.us-east-1.docdb.amazonaws.com
      - VALKEY_ENDPOINT=clustercfg.production-elasticache-us-east-1.hedavb.use1.cache.amazonaws.com
      - MSK_BROKERS=b-1.productionmskuseast1...kafka.us-east-1.amazonaws.com:9096
      - OPENSEARCH_ENDPOINT=https://vpc-production-os-use1-...us-east-1.es.amazonaws.comyaml
핵심 패턴Key Pattern {{metadata.labels.region}} 템플릿 변수 하나로 양 리전에 동일한 Application을 생성합니다. 리전별 엔드포인트 차이는 configMapGenerator의 literals에서 흡수하므로, 서비스 코드는 환경변수만 읽으면 됩니다.

멀티리전 아키텍처 설계 원칙Multi-Region Architecture Design Principles

Write-Primary / Read-Local 패턴과 설계 원칙Write-Primary / Read-Local Pattern & Design Principles Architecture

멀티리전 아키텍처에서 가장 어려운 문제는 데이터 일관성입니다. 모든 리전에서 동시에 쓰기를 허용하면 Last Write Wins 등의 충돌(conflict)이 발생하므로, 본 아키텍처는 Write-Primary / Read-Local 패턴을 채택합니다.

핵심 원칙

원칙설명구현
Write-Primary 쓰기 요청은 Primary 리전(us-east-1)으로 전달 Aurora Global Write Forwarding
Read-Local 읽기 요청은 가장 가까운 리전에서 처리 Route53 Latency-based Routing
Eventual Consistency 리전간 데이터 복제 지연 허용 (~1s 이내) 각 데이터스토어의 네이티브 복제
Active-Active (예외) DSQL은 양 리전에서 동시 쓰기 가능 DSQL Peer Linking (AWS 네이티브)

EC2 Sync Connector

On-Prem ↔ AWS CDC 동기화 아키텍처On-Prem ↔ AWS CDC Sync Architecture Database

On-Prem과 AWS 간의 데이터 동기화를 위해 EC2 기반 CDC Connector를 사용합니다. On-Prem DC와 AWS VPC를 Direct Connect로 연결하고, EC2 인스턴스에서 CDC 에이전트를 실행합니다.

동기화 구성

Source (On-Prem)Target (AWS)CDC 도구방향
PostgreSQL DSQL Debezium + Kafka Connect On-Prem → AWS (단방향)
MongoDB DocumentDB Mongo Change Stream + Custom Agent On-Prem → AWS (단방향)
Kafka MSK Kafka MirrorMaker2 양방향 (topic prefix)
Elasticsearch OpenSearch Logstash reindex pipeline On-Prem → AWS (단방향)
Redis / ElastiCache Redis(ElastiCache)는 캐시 용도이므로 데이터 동기화를 수행하지 않습니다. 각 리전에서 독립적으로 캐시를 워밍업하며, cache miss 시 DB에서 재로드합니다.

Direct Connect 구성

# Direct Connect 경로
On-Prem DC
    |--- Direct Connect 10Gbps ---> AWS us-west-2 VPC
                                        |
                                   EC2 Sync Connector
                                   (Debezium, MirrorMaker2, Custom Agent)
                                        |
                                   AWS Data Stores
                                   (DSQL, DocumentDB, MSK, OpenSearch)text

EC2 Sync Connector는 Step 6에서 제거됩니다. On-Prem 트래픽이 0%가 되면 CDC 동기화가 더 이상 필요하지 않으며, 이후에는 AWS 네이티브 리전간 동기화만 활성 상태로 유지됩니다.
