Multi-Region DR/Failover Strategy

Multi-Region DR/Failover Strategy Network

Current CCS status: NO DR capability — the prevailing mentality is "respond via OTA" after incidents occur. The AWS migration introduces Active-Active architecture with automatic failover via Route53 health checks.

Route53 Health Check + Latency Routing

terraform/modules/edge/route53/main.tf

# Health Check (per region)
resource "aws_route53_health_check" "regional" {
  for_each = var.lb_dns_names

  fqdn              = each.value
  port              = 80
  type              = "HTTP"
  resource_path     = var.health_check_path
  request_interval  = var.health_check_interval
  failure_threshold = var.health_check_failure_threshold

  tags = merge(var.tags, {
    Name   = "${var.environment}-${each.key}-health-check"
    Region = each.key
  })
}

# Latency-based Routing (per region)
resource "aws_route53_record" "api_latency" {
  for_each = var.lb_dns_names

  zone_id = var.zone_id
  name    = "api-internal.${data.aws_route53_zone.main.name}"
  type    = "A"

  alias {
    name                   = each.value
    zone_id                = var.lb_zone_ids[each.key]
    evaluate_target_health = true
  }

  set_identifier = each.key

  latency_routing_policy {
    region = each.key
  }

  health_check_id = aws_route53_health_check.regional[each.key].id
}

# CloudWatch Alarm for Health Check monitoring
resource "aws_cloudwatch_metric_alarm" "health_check" {
  for_each = var.lb_dns_names

  alarm_name          = "${var.environment}-${each.key}-health-check-alarm"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 2
  metric_name         = "HealthCheckStatus"
  namespace           = "AWS/Route53"
  period              = 60
  threshold           = 1
  treat_missing_data  = "breaching"
}hcl

Failover Sequence

# Automatic failover sequence (us-east-1 failure)

T+0s    Health Check fails (3 consecutive failures)
        |
T+30s   Route53 marks us-east-1 as unhealthy
        |
T+60s   DNS TTL expires, traffic routes to us-west-2
        |
        +---> Stateless services: Immediate recovery
        |
        +---> Aurora: Write forwarding to Primary
        |     (or promote Secondary if Primary region down)
        |
        +---> DSQL: No action needed (active-active)
        |
T+120s  Full traffic serving from us-west-2

# Manual actions (if Primary region down)
- Aurora: Initiate global database failover
- DocumentDB: Promote secondary cluster
- Update runbook status in PagerDutytext
Active-Active Benefits With Route53 latency-based routing, both regions actively serve traffic. Failover is seamless — users are simply routed to the remaining healthy region.

MongoDB I/O Bottleneck Migration

MongoDB I/O Bottleneck to AWS Migration Path Database

The current MongoDB deployment suffers from severe I/O bottlenecks due to legacy HDD storage, lack of host-level tuning, and 5x data volume increase from Gen1 to Gen3 vehicles.

Root Cause Analysis

FactorIssueImpact
StorageHDD residual (not SSD)IOPS bottleneck at ~500 IOPS
Host TuningNo FS/RAID/kernel optimizationSuboptimal I/O scheduling
Data GrowthGen3 vehicles 5x data vs Gen1Storage capacity exhaustion
ReplicationSingle replica setNo horizontal read scaling
