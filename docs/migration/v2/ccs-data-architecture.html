CCS Data Architecture

The CCDI (Connected Car Data Interface) is the data collection and processing layer for all vehicle telemetry. This section covers the current and target architecture, data tiering, Gen2/Gen3 processing differences, and personal information management.

CCDI Architecture

CCDI Architecture — AS-IS vs TO-BE Architecture

CCDI (Connected Car Data Interface) is the central data layer responsible for collecting, processing, and serving vehicle telemetry data. It handles all incoming signals from connected vehicles and distributes processed data to downstream services.

Data Tiering Strategy

Data Tiering Strategy — Hot/Warm/Cold Database

Data flows through three tiers based on access patterns and age. Two parallel lifecycles operate: DB retention (3 days) and S3 archive (permanent).

Tier Definitions

TierStorageRetentionLatencyUse Case
Hot ElastiCache (Redis) TTL: minutes P99 < 1ms VSS latest state, real-time queries
Warm MongoDB Atlas / DocumentDB 3-7 days P99 < 50ms Service data, recent history
Cold S3 + Athena Permanent Seconds (query) Historical analysis, compliance

Data Flow

Kafka Topic (VSS data)
       |
       +---> [Hot] Redis: SET vss:{vin} {latest_state} EX 300
       |
       +---> [Warm] MongoDB: insert with TTL index (3 days)
       |
       +---> [Cold] S3: Firehose delivery (Parquet format)
                    |
                    +---> Glue Catalog ---> Athena queries

# Two parallel lifecycles:
1. DB Retention:  Kafka -> Redis (5min) -> MongoDB (3 days) -> DELETE
2. S3 Archive:    Kafka -> Firehose -> S3 (permanent) -> Athena

# Note: History data currently OFF
# Plan to activate S3+Athena tier for compliance and analyticstext

Gen2 vs Gen3 Processing

Gen2 vs Gen3 — CCS 1.0 (TCP) vs CCS 2.0 (MQTT) Architecture

Two generations of connected car systems coexist with fundamentally different protocols and data volumes. VSS (Vehicle Signal Specification) is the largest and most critical dataset across both generations.

Generation Comparison

AspectGen2 (CCS 1.0)Gen3 (CCS 2.0)
ProtocolTCP (proprietary)MQTT (standard)
Fleet Size2-3M vehicles0.6M vehicles (growing)
Data per VehicleBaseline (1x)5x vs Gen1
Kafka Topicsgen2-vss-*, gen2-event-*gen3-vss-*, gen3-event-*
Schema FormatLegacy binaryProtobuf / Avro
ProcessingLegacy CCDIModernized CCDI

Data Volume Comparison

# Per-vehicle data rates
Gen2 vehicle:  ~50 KB/minute  (basic telemetry)
Gen3 vehicle:  ~250 KB/minute (rich sensors, ADAS data)

# Fleet total (current)
Gen2: 2.5M vehicles x 50 KB/min  = ~125 GB/min = ~180 TB/day
Gen3: 0.6M vehicles x 250 KB/min = ~150 GB/min = ~216 TB/day

# Projection (2026)
Gen2: Stable (legacy fleet)
Gen3: 2M vehicles x 250 KB/min = ~500 GB/min = ~720 TB/day

# Implication
Gen3 growth drives ALL capacity planning decisionstext

Processing Architecture

# Gen2 Path (Legacy)

# Gen3 Path (Modern)
Vehicle --[MQTT]--> IoT Core --> Kafka (gen3-*) --> CCDI v2 --> Tiered Storage

# Unified View
Both paths --> Kafka --> Unified CCDI (EKS) --> Common Data Laketext

VSS (Vehicle Signal Specification)

VSS is the standardized data model for vehicle signals. It represents the largest data volume and requires the lowest latency for real-time applications.

Personal Information Management

Personal Information (PI) & VIN Management Operations

Kafka messages contain VIN (Vehicle Identification Number) as the primary identifier. Whether VIN constitutes PI under US regulations requires legal confirmation, but the architecture must support PI-compliant data handling regardless.

VIN ID-Based Management Framework

# Proposed unified approach (Korea/US)

1. VIN Tokenization
   Raw VIN --> Tokenization Service --> Token ID
   Token ID used in all downstream processing
   VIN-to-Token mapping stored in secured vault

2. Data Segregation
   PI Data:     Separate Kafka topics (pi-*)
   Non-PI Data: Standard topics (telemetry-*)

3. Access Control
   PI topics:     IAM + SASL/SCRAM + audit logging
   Non-PI topics: Standard IAM authentication

# Migration note:
As long as VIN locations are identified in data flows,
migration proceeds without major blockers.

  - alias/ccs-rds-key      (Aurora encryption)
  - alias/ccs-msk-key      (MSK encryption)

us-west-2:
  - alias/ccs-ebs-key      (EBS encryption)
  - alias/ccs-s3-key       (S3 encryption)
  - alias/ccs-rds-key      (Aurora encryption)
  - alias/ccs-msk-key      (MSK encryption)

# Cross-region key policy
- S3 replication: destination uses destination-region key
- Aurora Global: each region uses own KMS key
- No key material crosses regionstext
