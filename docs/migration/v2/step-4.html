Step 4: 트래픽 분산 시작Traffic Distribution Start (7:1.5:1.5)

CloudFront Origin Group을 통해 AWS 리전으로 트래픽 분산을 시작합니다. Route53 latency-based routing과 Health Check로 리전간 failover를 보장합니다.

트래픽 분배Traffic Distribution On-Prem 70% / us-west-2 15% / us-east-1 15% — AWS 리전으로 30% 트래픽 전환

Latency-based 라우팅Routing

Route53 Latency-based Routing + Health Check Network

Route53의 Latency-based Routing은 사용자 위치에서 가장 낮은 지연 시간을 가진 리전으로 DNS를 라우팅합니다. 각 리전의 NLB에 대해 Health Check를 설정하여, 한 리전이 다운되면 자동으로 다른 리전으로 failover됩니다.

Route53 Terraform 정의

terraform/modules/edge/route53/main.tf

# Health Check (리전별)
resource "aws_route53_health_check" "regional" {
  for_each = var.lb_dns_names

  fqdn              = each.value
  port              = 80
  type              = "HTTP"
  resource_path     = var.health_check_path
  request_interval  = var.health_check_interval
  failure_threshold = var.health_check_failure_threshold

  tags = merge(var.tags, {
    Name   = "${var.environment}-${each.key}-health-check"
    Region = each.key
  })
}

# Latency-based Routing (리전별)
resource "aws_route53_record" "api_latency" {
  for_each = var.lb_dns_names

  zone_id = var.zone_id
  name    = "api-internal.${data.aws_route53_zone.main.name}"
  type    = "A"

  alias {
    name                   = each.value
    zone_id                = var.lb_zone_ids[each.key]
    evaluate_target_health = true
  }

  set_identifier = each.key

  latency_routing_policy {
    region = each.key
  }

  health_check_id = aws_route53_health_check.regional[each.key].id
}

# CloudWatch Alarm (Health Check 모니터링)
resource "aws_cloudwatch_metric_alarm" "health_check" {
  for_each = var.lb_dns_names

  alarm_name          = "${var.environment}-${each.key}-health-check-alarm"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 2
  metric_name         = "HealthCheckStatus"
  namespace           = "AWS/Route53"
  period              = 60
  threshold           = 1
  treat_missing_data  = "breaching"
}hcl
for_each 패턴 for_each = var.lb_dns_names로 리전별 Health Check, DNS Record, CloudWatch Alarm을 한번에 생성합니다. 새 리전을 추가할 때 변수에 엔트리만 추가하면 모든 리소스가 자동으로 생성됩니다.

리전간 Failover 메커니즘Cross-Region Failover Mechanism

Aurora Global Write Forwarding + Route53 Failover Network

리전 장애 시 두 계층에서 failover가 발생합니다: (1) DNS 레벨에서 Route53 Health Check가 장애 리전을 감지하고 트래픽을 전환하고, (2) 데이터 레벨에서 Aurora Global Database가 Secondary를 Primary로 승격합니다.

Aurora Global Write Forwarding

Step 3에서 리전간 동기화를 활성화했지만, Aurora의 Write Forwarding은 실제 트래픽이 양 리전에 분산되는 이 단계에서 동작하기 시작합니다.

terraform/modules/data/aurora-global/main.tf (발췌)

# Secondary cluster: Write Forwarding 활성화
source_region                  = var.is_primary ? null : var.source_region
enable_global_write_forwarding = var.is_primary ? null : var.enable_write_forwardinghcl

Failover 시나리오

시나리오감지 시간복구 방법데이터 손실
us-east-1 장애 ~60s (Health Check) Route53가 us-west-2로 자동 전환 없음 (DSQL Active-Active)
us-west-2 장애 ~60s Route53가 us-east-1로 자동 전환 없음 (DSQL Active-Active)
Aurora Primary 장애 ~120s Global Database failover (Secondary 승격) RPO (Recovery Point Objective) < 1s
단일 AZ 장애 ~30s Multi-AZ 자동 전환 (리전 내) 없음
Write Forwarding 주의 Aurora의 enable_global_write_forwarding은 Secondary에서의 쓰기 요청을 Primary로 전달합니다. 추가 지연 (~100ms cross-region)이 발생하므로, 쓰기 빈도가 높은 서비스는 Primary 리전에서 직접 처리하는 것이 좋습니다. DSQL은 이 제약이 없어 양 리전에서 동등하게 쓰기 가능합니다.

Canary 배포 전략Deployment Strategy

점진적 트래픽 전환과 롤백 전략Gradual Traffic Shift & Rollback Strategy Network

On-Prem에서 AWS로의 트래픽 전환은 Canary 방식으로 점진적으로 수행합니다. 각 단계에서 에러율과 지연 시간을 모니터링하고, 문제 발생 시 즉시 이전 단계로 롤백합니다.

롤백 기준 (자동)

# CloudWatch Alarm 기반 롤백 트리거
Error Rate   > 1%     # 5분간 에러율 1% 초과
P99 (99번째 백분위수) Latency  > 3s     # P99 응답 시간 3초 초과
Health Check  FAIL    # Route53 Health Check 실패
Data Lag     > 30s    # 리전간 데이터 동기화 지연 30초 초과
