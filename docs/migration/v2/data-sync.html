CCDI 데이터 동기화CCDI Data Synchronization

CCDI(Connected Car Data Integration)의 핵심은 차량 데이터의 허용 지연(SLA)을 정의하는 것입니다. VSS(Vehicle Signal Specification) 데이터 유형별로 다른 복제 전략을 적용합니다.The core of CCDI (Connected Car Data Integration) is defining the allowable lag (SLA) for vehicle data. Different replication strategies are applied based on VSS (Vehicle Signal Specification) data types.

CCDI Replication Lag SLA

CCDI 데이터 유형별 허용 지연과 모니터링CCDI Per-Data-Type Allowable Lag and Monitoring Database

CCDI는 차량에서 수집되는 다양한 신호(VSS)를 처리합니다. 데이터 유형별 중요도와 허용 지연이 다르며, 이에 맞는 복제 전략을 선택합니다.CCDI processes various signals (VSS) collected from vehicles. Each data type has different importance and allowable lag, and we select appropriate replication strategies accordingly.

Lag Spike 대응Lag Spike Response VSS 복제 지연이 1초를 초과하면 차량 상태 조회에 영향을 줍니다. 원인(네트워크, 대량 write, DB 리소스)을 파악하고, 필요 시 트래픽을 Primary 리전으로 집중시키는 긴급 대응이 필요합니다.When VSS replication lag exceeds 1 second, vehicle status queries are affected. Identify the cause (network, bulk writes, DB resources) and implement emergency response by concentrating traffic to the Primary region if needed.

CCS Data Conflict Resolution

CCDI 충돌 방지 설계와 해결 전략CCDI Conflict Prevention Design and Resolution Strategy Database

CCDI의 차량 텔레메트리 데이터는 본질적으로 append-only입니다. 차량이 데이터를 생성하고 서버가 수신하는 단방향 흐름이므로 충돌이 거의 발생하지 않습니다.CCDI vehicle telemetry data is inherently append-only. Since vehicles generate data and servers receive it in a one-way flow, conflicts rarely occur.

CCDI Append-Only Pattern

# Vehicle telemetry is write-once
# No UPDATE, only INSERT

INSERT INTO vss_data (
    vin,           # Vehicle Identification Number
    timestamp,     # Immutable event time
    signal_type,
    signal_value
) VALUES (?, ?, ?, ?);

# Primary key: (vin, timestamp, signal_type)
# Same event from same vehicle = same key = idempotenttext

VIN-based Partitioning (Single-Writer Guarantee)

# VIN determines primary region
# Hash(VIN) % 2 → region affinity

VIN: KMHXX00XXXX000001 → hash(0x7A3F...) % 2 = 0 → us-east-1
VIN: KMHXX00XXXX000002 → hash(0x8B2E...) % 2 = 1 → us-west-2

# Benefits:
# 1. Single writer per VIN → no conflicts
# 2. All data for a vehicle in one region → no cross-region joins
# 3. Predictable failover (VIN→region mapping is deterministic)text

CCDI Initial Data Migration Strategy

CCDI 초기 데이터 이관 계획과 Korea CCS 선례CCDI Initial Data Migration Plan and Korea CCS Precedent Operations

CCDI 초기 이관은 Full snapshot + CDC catch-up + Validation 3단계로 진행합니다. Korea CCS의 2025.06 Kafka→MSK 마이그레이션 경험을 반영합니다.CCDI initial migration proceeds in 3 phases: Full snapshot + CDC catch-up + Validation. This incorporates lessons from Korea CCS's 2025.06 Kafka→MSK migration.

Phase Approach

# Phase 1: Full Snapshot (서비스 운영 중)
mongodump --oplog → S3 → mongorestore    # ~200GB, 2-3시간
MirrorMaker2 initial sync                 # ~1TB, 1-2시간

# Phase 2: CDC Catch-up (스냅샷 이후 변경분)
MongoDB Change Stream → DocumentDB        # oplog 기반 tailing
MirrorMaker2 continuous                   # offset 기반 sync

# Phase 3: Validation
Document count comparison                 # collection별 count 일치
Sample document hash                      # 랜덤 샘플 해시 비교
API response comparison                   # 주요 API 응답 diff

# Phase 4: Cutover
DNS switch (Route53 weighted → 100%)      # ~60s propagation
MirrorMaker2 reverse (new primary→old)    # 롤백 대비text
