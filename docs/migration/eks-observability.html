EKS ObservabilityEKS Observability

On-Prem Nagios/Zabbix에서 클라우드 네이티브 관측성 스택으로 전환합니다. Metrics(Prometheus), Traces(OTel+Tempo), Logs(FluentBit+CloudWatch)의 3 Pillars를 통합하여 마이크로서비스를 모니터링합니다.Transition from On-Prem Nagios/Zabbix to cloud-native observability stack. Monitor microservices by integrating 3 Pillars: Metrics (Prometheus), Traces (OTel+Tempo), and Logs (FluentBit+CloudWatch).

관측성 스택 개요Observability Stack Overview

클라우드 네이티브 환경에서는 메트릭, 트레이스, 로그의 3가지 신호를 통합하여 시스템을 관측합니다. 각 신호는 서로 다른 관점에서 정보를 제공하며, 상관관계 분석을 통해 문제를 빠르게 진단합니다.In cloud-native environments, observe systems by integrating 3 signals: metrics, traces, and logs. Each signal provides information from different perspectives, enabling fast diagnosis through correlation analysis.

Pillar 도구Tools 수집 방식Collection Method 저장소Storage 용도Purpose

On-Prem vs EKS 모니터링 비교On-Prem vs EKS Monitoring Comparison On-Prem에서는 Nagios/Zabbix로 호스트 기반 모니터링을 했지만, EKS에서는 Pod가 동적으로 생성/삭제됩니다. 서비스 디스커버리 기반의 Prometheus와 분산 트레이싱이 필수이며, 로그도 중앙 집중화해야 합니다.On-Prem used host-based monitoring with Nagios/Zabbix, but in EKS, Pods are dynamically created/deleted. Service discovery-based Prometheus and distributed tracing are essential, and logs must be centralized.

메트릭 파이프라인Metrics Pipeline

Prometheus + Grafana 메트릭 수집Prometheus + Grafana Metrics Collection Observability

Prometheus는 Pull 기반으로 타겟의 /metrics 엔드포인트를 스크래핑하여 시계열 데이터를 수집합니다. Kubernetes 환경에서는 ServiceMonitor/PodMonitor CRD로 스크래핑 대상을 선언적으로 관리합니다.Prometheus collects time-series data by scraping target's /metrics endpoints using Pull model. In Kubernetes, ServiceMonitor/PodMonitor CRDs declaratively manage scraping targets.

AMP (Amazon Managed Prometheus) 옵션AMP (Amazon Managed Prometheus) Option

장기 보존(150일)과 운영 부담 최소화가 필요하면 AMP를 사용합니다. Prometheus는 remote_write로 AMP에 메트릭을 전송합니다.Use AMP for long-term retention (150 days) and minimal operational overhead. Prometheus sends metrics to AMP via remote_write.

prometheus:
  prometheusSpec:
    remoteWrite:
      - url: https://aps-workspaces.us-east-1.amazonaws.com/workspaces/ws-xxx/api/v1/remote_write
        sigv4:
          region: us-east-1
        queueConfig:
          maxSamplesPerSend: 1000
          maxShards: 200
          capacity: 2500yaml

분산 트레이싱Distributed Tracing

OTel Collector + Tempo 분산 트레이싱OTel Collector + Tempo Distributed Tracing Observability

마이크로서비스 간 호출을 추적하여 레이턴시 병목을 찾습니다. OTel Collector가 애플리케이션에서 트레이스를 수집하고, Tempo(S3 백엔드)와 X-Ray에 동시 전송합니다.Track calls between microservices to find latency bottlenecks. OTel Collector collects traces from applications and sends them to both Tempo (S3 backend) and X-Ray.

OTel Collector DaemonSet 구성OTel Collector DaemonSet Configuration

각 노드에서 트레이스를 수집하여 Tempo와 X-Ray로 듀얼 export합니다.Collect traces from each node and dual export to Tempo and X-Ray.

exporters:
  otlp/tempo:
    endpoint: tempo-distributor.tempo.svc:4317
    tls:
      insecure: true
  awsxray:
    region: us-east-1
    indexed: true

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, batch, tail_sampling]
      exporters: [otlp/tempo, awsxray]yaml

Tail-based Sampling 전략Tail-based Sampling Strategy

모든 트레이스를 저장하면 비용이 급증합니다. 에러와 느린 요청은 100% 샘플링하고, 정상 요청은 10%만 샘플링합니다.Storing all traces causes cost explosion. Sample 100% of errors and slow requests, but only 10% of normal requests.

processors:
  tail_sampling:
    decision_wait: 30s
    num_traces: 100000
    policies:
      # Sample 100% of errors
      - name: errors-policy
        type: status_code
        status_code:
          status_codes: [ERROR]
      # 500ms 이상 느린 요청 100% 샘플링
      - name: latency-policy
        type: latency
        latency:
          threshold_ms: 500
      # Sample 10% of remaining
      - name: probabilistic-policy
        type: probabilistic
        probabilistic:
          sampling_percentage: 10yaml

로그 수집Log Collection

FluentBit + CloudWatch Logs 로그 파이프라인FluentBit + CloudWatch Logs Pipeline Observability

FluentBit DaemonSet이 각 노드에서 컨테이너 로그를 수집하여 CloudWatch Logs로 전송합니다. 경량(~10MB 메모리)이면서 다양한 출력 플러그인을 지원합니다.FluentBit DaemonSet collects container logs from each node and sends them to CloudWatch Logs. Lightweight (~10MB memory) while supporting various output plugins.

알림Alerting

Alertmanager + SNS 알림 파이프라인Alertmanager + SNS Alert Pipeline Observability

Prometheus Alertmanager가 알림을 그룹화, 라우팅, 억제합니다. SNS를 통해 PagerDuty, Slack, 이메일로 알림을 전달합니다.Prometheus Alertmanager groups, routes, and suppresses alerts. Deliver alerts to PagerDuty, Slack, and email via SNS.

SLO 기반 알림 규칙SLO-based Alert Rules

에러율과 레이턴시 P99를 기준으로 SLO 위반 시 알림을 발생시킵니다.Trigger alerts when SLO is violated based on error rate and latency P99.

apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: slo-alerts
  namespace: monitoring
spec:
  groups:
    - name: slo.rules
      rules:
        # Alert when error rate exceeds 1%
        - alert: HighErrorRate
          expr: |
            sum(rate(http_requests_total{status=~"5.."}[5m])) by (service)
            / sum(rate(http_requests_total[5m])) by (service) > 0.01
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "High error rate on {{ $labels.service }}"
            description: "Error rate is {{ printf \"%.2f\" $value }}%"

        # P99 레이턴시 2초 초과 시 알림
        - alert: HighLatencyP99
          expr: |
            histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service)) > 2
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "High P99 latency on {{ $labels.service }}"yaml

Alert Fatigue 방지Preventing Alert Fatigue 너무 많은 알림은 무시로 이어집니다. 심각도를 명확히 분류하고, 중복 알림은 그룹화하세요. Critical은 즉시 대응이 필요한 경우만, Warning은 근무 시간 내 대응, Info는 대시보드에서만 확인합니다.Too many alerts lead to being ignored. Clearly classify severity and group duplicate alerts. Critical only for immediate response, Warning for business hours response, Info only visible in dashboards.
