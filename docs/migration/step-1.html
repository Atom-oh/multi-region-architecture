Step 1: us-west-2 배포 (인프라 준비)Deploy (Infra Ready)

us-west-2 리전에 전체 인프라를 배포합니다. 아직 트래픽은 전달하지 않으며, 동기화 완료 후 Step 3에서 트래픽을 시작합니다. On-Prem DC가 us-west 지역에 위치하므로, 네트워크 지연이 적은 us-west-2를 먼저 배포하여 검증합니다.Deploy the full infrastructure to the us-west-2 region. No traffic is routed yet — traffic starts in Step 3 after sync is confirmed. Since the On-Prem DC is located in the us-west region, we deploy to us-west-2 first for validation due to lower network latency.

트래픽 분배Traffic Distribution On-Prem 100% — 인프라 배포만, 트래픽 미유입Infrastructure deployment only, no traffic

ArgoCD App of Apps 패턴Pattern

ArgoCD ApplicationSet + Kustomize 오버레이ArgoCD ApplicationSet + Kustomize Overlays GitOps

ArgoCD ApplicationSet의 clusters generator를 사용하여, 하나의 ApplicationSet 정의로 양 리전에 동일한 서비스를 자동 배포합니다. 리전별 차이(엔드포인트, SG ID, 인증서 ARN 등)는 Kustomize 오버레이에서 처리합니다.Using the clusters generator of ArgoCD ApplicationSet, a single ApplicationSet definition automatically deploys the same services to both regions. Regional differences (endpoints, SG IDs, certificate ARNs, etc.) are handled in Kustomize overlays.

ApplicationSet: Core 서비스ApplicationSet: Core Services

k8s/infra/argocd/apps/appset-core.yaml

apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: core-services
  namespace: argocd
spec:
  generators:
    - clusters:
        selector:
          matchExpressions:
            - key: region
              operator: Exists
  template:
    metadata:
      name: 'core-{{metadata.labels.region}}'
    spec:
      project: default
      source:
        repoURL: git@github.com:Atom-oh/multi-region-architecture.git
        targetRevision: main
        path: 'k8s/overlays/{{metadata.labels.region}}/core'
      destination:
        server: '{{server}}'
        namespace: core-services
      syncPolicy:
        automated:
          prune: true
          selfHeal: trueyaml

Kustomize: 리전별 ConfigMap 오버레이Kustomize: Regional ConfigMap Overlays

k8s/overlays/us-east-1/kustomization.yaml (발췌)

configMapGenerator:
  - name: region-config
    namespace: platform
    literals:
      - REGION=us-east-1
      - REGION_ROLE=PRIMARY
      - MONGODB_ATLAS_URI=mongodb+srv://production-atlas.xxxxx.mongodb.net
      - VALKEY_ENDPOINT=clustercfg.production-elasticache-us-east-1.xxxxxx.use1.cache.amazonaws.com
      - MSK_BROKERS=b-1.productionmskuseast1...kafka.us-east-1.amazonaws.com:9096
      - OPENSEARCH_ENDPOINT=https://vpc-production-os-use1-...us-east-1.es.amazonaws.comyaml
핵심 패턴Key Pattern {{metadata.labels.region}} 템플릿 변수 하나로 양 리전에 동일한 Application을 생성합니다. 리전별 엔드포인트 차이는 configMapGenerator의 literals에서 흡수하므로, 서비스 코드는 환경변수만 읽으면 됩니다.A single {{metadata.labels.region}} template variable creates identical Applications in both regions. Regional endpoint differences are absorbed in configMapGenerator's literals, so service code only needs to read environment variables.

멀티리전 아키텍처 설계 원칙Multi-Region Architecture Design Principles

Write-Primary / Read-Local 패턴과 설계 원칙Write-Primary / Read-Local Pattern & Design Principles Architecture

멀티리전 아키텍처에서 가장 어려운 문제는 데이터 일관성입니다. 모든 리전에서 동시에 쓰기를 허용하면 Last Write Wins 등의 충돌(conflict)이 발생하므로, 본 아키텍처는 Write-Primary / Read-Local 패턴을 채택합니다.The most challenging problem in multi-region architecture is data consistency. Allowing simultaneous writes from all regions causes conflicts like Last Write Wins, so this architecture adopts the Write-Primary / Read-Local pattern.
