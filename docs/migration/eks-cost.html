EKS 비용 최적화EKS Cost Optimization

On-Prem 고정 CapEx 모델에서 AWS 종량제(OpEx) 모델로 전환하면서 FinOps 문화를 도입합니다. Spot 인스턴스, Graviton, 리소스 최적화를 통해 컴퓨팅 비용을 최대 70%까지 절감할 수 있습니다.Introduce FinOps culture as you transition from On-Prem fixed CapEx model to AWS pay-as-you-go (OpEx) model. Reduce compute costs by up to 70% through Spot instances, Graviton, and resource optimization.

FinOps 원칙FinOps Principles

FinOps: 클라우드 비용 최적화 문화FinOps: Cloud Cost Optimization Culture Operations

FinOps는 Inform(가시성 확보) → Optimize(최적화 실행) → Operate(운영 및 거버넌스) 사이클을 통해 지속적으로 클라우드 비용을 최적화하는 운영 모델입니다.FinOps is an operational model that continuously optimizes cloud costs through the Inform (visibility) → Optimize (execution) → Operate (governance) cycle.

Spot 인스턴스 전략Spot Instance Strategy

Karpenter + Spot으로 최대 90% 비용 절감Up to 90% Cost Savings with Karpenter + Spot Operations

Spot 인스턴스는 On-Demand 대비 최대 90% 저렴하지만, 2분 경고 후 중단될 수 있습니다. Karpenter의 혼합 배치 전략과 인터럽트 핸들링으로 안정적으로 Spot을 활용합니다.Spot instances are up to 90% cheaper than On-Demand but can be interrupted with 2-minute warning. Use Spot reliably with Karpenter's mixed allocation strategy and interrupt handling.

Karpenter Spot NodePool 설정Karpenter Spot NodePool Configuration

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: worker-tier
spec:
  template:
    metadata:
      labels:
        node-pool: worker-tier
    spec:
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
      requirements:
        # Spot preferred, On-Demand fallback
        - key: karpenter.sh/capacity-type
          operator: In
          values:
            - spot
            - on-demand
        # Distribute interruptions across diverse instance types
        - key: karpenter.k8s.aws/instance-family
          operator: In
          values:
            - m6i
            - m7i
            - c6i
            - c7i
            - r6i
        - key: karpenter.k8s.aws/instance-size
          operator: In
          values:
            - large
            - xlarge
            - 2xlarge
  weight: 50
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 30syaml

Stateful 워크로드 주의Stateful Workload Caution 데이터베이스, 캐시, StatefulSet 워크로드에는 Spot을 사용하지 마세요. 2분 경고 시간 내에 데이터 복제가 완료되지 않으면 데이터 손실이 발생할 수 있습니다.Do not use Spot for databases, caches, or StatefulSet workloads. Data loss can occur if data replication is not completed within the 2-minute warning time.

Graviton (ARM) 활용Graviton (ARM) Utilization

Graviton으로 비용 대비 성능 20% 향상20% Better Price-Performance with Graviton Operations

AWS Graviton (ARM64) 프로세서는 x86 대비 동일 비용에서 약 20% 더 나은 성능을 제공합니다. Go, Java, Python 등 대부분의 언어에서 ARM64 빌드가 가능하며, 멀티 아키텍처 이미지로 점진적으로 전환할 수 있습니다.AWS Graviton (ARM64) processors provide about 20% better performance at the same cost compared to x86. ARM64 builds are possible in most languages like Go, Java, Python, and you can gradually transition with multi-architecture images.

멀티 아키텍처 빌드Multi-Architecture Build

# Build multi-architecture images with Docker buildx
docker buildx create --name multiarch --use
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --tag 123456789012.dkr.ecr.us-east-1.amazonaws.com/api-server:latest \
  --push .bash

  name: general
spec:
  template:
    spec:
      requirements:
        # Allow both x86 and ARM

          operator: In

            - amd64
            - arm64
        # Include Graviton instance types
        - key: karpenter.k8s.aws/instance-family
          operator: In
          values:
            - m6i     # x86
            - m6g     # Graviton

            - r6g     # Gravitonyaml

For Python/Node.js, verify that native C extensions (numpy, pandas, etc.) are built for ARM64.

리소스 최적화Resource Optimization

VPA + Goldilocks로 Right-SizingRight-Sizing with VPA + Goldilocks Operations

과다 프로비저닝된 리소스는 비용 낭비의 주요 원인입니다. VPA(Vertical Pod Autoscaler) 권장값을 기반으로 Request/Limit을 최적화합니다.Over-provisioned resources are a major cause of cost waste. Optimize Request/Limit based on VPA (Vertical Pod Autoscaler) recommendations.

VPA 권장값 확인Check VPA Recommendations

apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: api-server-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  updatePolicy:
    updateMode: "Off"  # Recommendation only, no auto-applyyaml
# Query VPA recommendations
kubectl describe vpa api-server-vpa

# Example output:
# Recommendation:
#   Container Recommendations:
#     Target:          cpu: 250m, memory: 512Mi  (recommended request)
#     Lower Bound:     cpu: 100m, memory: 256Mi
#     Upper Bound:     cpu: 500m, memory: 1Gi    (recommended limit)bash

LimitRange + ResourceQuota로 거버넌스Governance with LimitRange + ResourceQuota

apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: production
spec:
  limits:
    - type: Container
      default:           # Default limit
        cpu: "500m"
        memory: "512Mi"
      defaultRequest:   # Default request
        cpu: "100m"
        memory: "128Mi"
      max:              # Maximum allowed
        cpu: "4"
        memory: "8Gi"
---
apiVersion: v1
kind: ResourceQuota
metadata:
  name: namespace-quota
  namespace: production
spec:
  hard:
    requests.cpu: "20"
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
    pods: "50"yaml
Right-Sizing 권장 접근법Recommended Right-Sizing Approach Request: VPA Target 값 또는 P70 실제 사용량. Limit: VPA Upper Bound 또는 P99 + 20% 여유. CPU Limit은 latency-sensitive 워크로드에서 throttling을 유발할 수 있으므로 제거를 고려하세요.Request: VPA Target value or P70 actual usage. Limit: VPA Upper Bound or P99 + 20% headroom. Consider removing CPU Limit for latency-sensitive workloads as it can cause throttling.
