CCS Multi-Region 아키텍처CCS Multi-Region Architecture

Connected Car Service(CCS)의 멀티리전 아키텍처는 차량 데이터의 특성을 반영합니다. 서비스별 CAP 선택, 장애 격리 전략, 그리고 차량 텔레메트리 데이터 전송 비용을 중심으로 설계합니다.The multi-region architecture for Connected Car Service (CCS) reflects the characteristics of vehicle data. We design around per-service CAP selection, failure isolation strategies, and vehicle telemetry data transfer costs.

CAP Theorem Applied to CCS Services

CCS 서비스별 CAP 선택과 PACELC 확장CCS Per-Service CAP Selection and PACELC Extension Architecture

CCS는 차량 데이터(CCDI), 내비게이션(GIS), 콘텐츠(CPX), 연결 관리(ServiceHub), OTA 업데이트(CUS) 등 다양한 서비스로 구성됩니다. 각 서비스의 데이터 특성에 따라 CP vs AP 선택이 달라집니다.CCS consists of various services including vehicle data (CCDI), navigation (GIS), content (CPX), connection management (ServiceHub), and OTA updates (CUS). The CP vs AP choice varies based on each service's data characteristics.

CCS를 위한 PACELC 확장PACELC Extension for CCS

# PACELC (Partition→AC, Else→LC) — Extension of CAP Theorem
# On Partition: A vs C, Normal: Latency vs Consistency
# Cross-region RTT: ~60ms (us-east-1 ↔ us-west-2)

CCDI:       P→C, E→C   # Always consistency first (vehicle data integrity)
GIS:        P→A, E→L   # Latency first even in normal state (real-time navigation)
CPX:        P→A, E→L   # Stale content allowed, use CDN cache
ServiceHub: P→C, E→L   # Latency in normal, consistency on failure
CUS:        P→A, E→L   # OTA allows delay, only guarantee eventual deliverytext
Cross-Region Latency Budget 정상 상태에서 리전간 RTT ~60ms. CCDI Write Forwarding 추가 시 총 ~80-100ms. GIS 경로 탐색은 이 지연을 허용할 수 없어 리전별 독립 캐시를 사용합니다.Cross-region RTT is ~60ms under normal conditions. With CCDI Write Forwarding, total latency is ~80-100ms. GIS route search cannot tolerate this delay, so we use independent caches per region.

Blast Radius per CCS Service

CCS 서비스별 장애 영향 범위와 격리 전략CCS Per-Service Blast Radius and Isolation Strategy Architecture

CCS 멀티리전의 핵심 가치는 차량 서비스 연속성입니다. 한 리전 장애가 다른 리전의 차량에 영향을 주지 않도록 격리 계층을 설계합니다.The core value of CCS multi-region is vehicle service continuity. We design isolation layers to ensure a failure in one region does not affect vehicles in another region.

격리 계층 (Isolation Layers)Isolation Layers

# Failure Isolation Layers (inner → outer)

L1: Pod-level        # Pod crash → K8s auto-restart (~5s)
L2: Node-level       # Node failure → Karpenter replacement node (~60s)
L3: AZ-level         # AZ failure → Multi-AZ auto-failover
L4: Region-level     # Region failure → Route53 failover (~60-120s)
L5: Global-level     # CloudFront/WAF → Global edge defensetext

CCS Multi-Region Cost Structure

CCS 멀티리전 비용 구조와 최적화 전략CCS Multi-Region Cost Structure and Optimization Strategy Operations

CCS 멀티리전은 단일 리전 대비 ~1.8-2.2x 비용이 발생합니다. 특히 차량 텔레메트리 데이터 전송량(10TB/day)과 MSK 복제(50MB/s)가 주요 비용 요소입니다.CCS multi-region costs ~1.8-2.2x compared to single region. Vehicle telemetry data transfer (10TB/day) and MSK replication (50MB/s) are the main cost drivers.

핵심 절감 전략Key Cost Reduction Strategies

# 1. Karpenter Spot (40% compute savings, additional savings possible with Graviton)
capacity-type: [spot, on-demand]
arch: [amd64]   # Currently amd64 only, arm64 (Graviton) can be added

# 2. Independent ElastiCache Operation (Eliminates Cross-Region Transfer)
# GIS route cache is independent per region → Zero replication cost

# 3. Tail-based Sampling (90% observability cost reduction)
errors:        100%   # Store all errors
slow (>500ms): 100%   # Store all slow requests
default:       10%    # Only 10% of normal traces

# 4. MSK Topic-level Replication
# Replicate only cross-region essential topics, not all topics
replicated:    ccdi.vss, ccdi.location  # Required
local-only:    analytics.*, logs.*      # Excluded from replicationtext
