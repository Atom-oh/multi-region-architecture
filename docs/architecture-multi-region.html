Multi-Region Architecture

AWS 기반 멀티리전 쇼핑몰 플랫폼 — Write-Primary / Read-Local 패턴

ⓘ Executive Summary

본 아키텍처는 AWS의 두 리전(us-east-1 Primary, us-west-2 Secondary)에 걸쳐 운영되는 쇼핑몰 플랫폼입니다. Write-Primary / Read-Local 패턴을 채택하여 쓰기는 Primary 리전으로 포워딩하고, 읽기는 가장 가까운 리전에서 처리합니다.

20개 마이크로서비스가 5개 도메인(Core, User, Fulfillment, Business, Platform)으로 조직되며, Go/Gin, Java/Spring Boot, Python/FastAPI 세 가지 기술 스택으로 구현됩니다. 모든 트래픽은 CloudFront → Route53 (Latency-based) → NLB → EKS 경로로 흐르며, 직접 ALB 접근은 차단됩니다.

Design Principles

⚙ Write-Primary / Read-Local

모든 쓰기 요청은 us-east-1(Primary)에서 처리. 읽기는 각 리전의 로컬 복제본 사용. Aurora Global Write Forwarding으로 Secondary에서도 쓰기 가능 (forwarded).

🔒 CloudFront-First Security

모든 퍼블릭 트래픽은 반드시 CloudFront를 통과. ALB/NLB 보안 그룹은 CloudFront managed prefix list로만 제한. 0.0.0.0/0 인바운드 규칙 완전 금지.

🛠 GitOps with ArgoCD

모든 K8s 리소스는 ArgoCD ApplicationSet으로 관리. 리전별 Kustomize overlay로 환경 분리. Git 커밋이 배포의 single source of truth.

📊 Unified Observability

OTel Collector가 traces/logs/metrics를 통합 수집. ClickHouse + Tempo + Prometheus 3중 백엔드. Grafana에서 metric → trace → log 연결 (exemplar).

Traffic Flow

그림 1. End-to-end traffic flow: User → CloudFront → Route53 (Latency) → NLB → API Gateway → Services
핵심 규칙: 모든 퍼블릭 트래픽은 반드시 CloudFront를 경유합니다. NLB 보안 그룹은 CloudFront managed prefix list(com.amazonaws.global.cloudfront.origin-facing)만 허용하며, 포트 범위 80-443의 단일 규칙을 사용합니다 (prefix list 45엔트리 × 규칙 한도 60의 제약).

Known Gaps (Deep Research Audit 결과):
  • us-west-2는 사실상 read-only (완전한 active-active 아님)
  • MSK에 cross-region replication 미설정 (unbounded RPO)
  • CloudFront에 origin failover group 미설정
  • 현재 RTO: ~15-20분 (목표 <10분)
  • Circuit breaker 미구현
